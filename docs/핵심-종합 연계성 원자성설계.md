# 핵심-종합 연계성 원자성 설계서

> **버전**: 1.0 (2026-01-31)  
> **목적**: CP/PFD/PFMEA 원자성 DB 저장 규칙의 핵심 설계  
> **중요도**: ⭐⭐⭐⭐⭐ (앱의 핵심 로직 - 원자핵)

---

## 📌 핵심 원칙

```
┌─────────────────────────────────────────────────────────────────┐
│     컬럼명 = DB필드명 = 코드 key = 100% 완전 일치              │
│     이 원칙이 깨지면 → 연동 불가 → 렌더링 오류                 │
└─────────────────────────────────────────────────────────────────┘
```

## 🔴 종합 원자성 DB 아키텍처 (핵심!)

### 문제: 분산 DB 구조

```
❌ 현재 (문제 구조):
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ ControlPlan │  │   PfdItem   │  │  PFMEA L3   │
│    Item     │  │             │  │   Items     │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       ▼                ▼                ▼
   [CP 화면]        [PFD 화면]       [FMEA 화면]
   
   → 각 문서별 DB가 분리되어 연계성 확보 어려움
   → 동기화 로직이 복잡해지고 불일치 발생
```

### 해결: 종합 연계성 DB (Single Source of Truth)

```
✅ 목표 (종합 DB 구조):
                    ┌─────────────────────────────┐
                    │    UnifiedProcessItem       │
                    │      (종합 원자성 DB)        │
                    │                             │
                    │  - processNo, processName   │
                    │  - processDesc, partName    │
                    │  - equipment, workElement   │
                    │  - productChar, processChar │
                    │  - specialChar, etc...      │
                    └──────────────┬──────────────┘
                                   │
           ┌───────────────────────┼───────────────────────┐
           ▼                       ▼                       ▼
    ┌─────────────┐         ┌─────────────┐         ┌─────────────┐
    │   CP View   │         │  PFD View   │         │  FMEA View  │
    │  (관리방법)  │         │  (공정흐름)  │         │  (위험분석)  │
    └─────────────┘         └─────────────┘         └─────────────┘
    
    → 종합 DB에서 공통 데이터 제공
    → 각 문서는 View 역할 (문서별 고유 필드만 추가)
    → 연계성 100% 보장
```

### 종합 DB 필드 분류

| 분류 | 필드 | 저장 위치 | 연동 |
|------|------|----------|------|
| **공통 필드** | processNo, processName, processLevel, processDesc | 종합 DB | CP, PFD, FMEA 공유 |
| **공통 필드** | partName, equipment | 종합 DB | CP, PFD 공유 |
| **공통 필드** | productChar, processChar, specialChar | 종합 DB | CP, PFD, FMEA 공유 |
| **FMEA 전용** | workElement (4M+1E) | 종합 DB | FMEA, CP 공유 |
| **FMEA 전용** | failureMode, failureEffect, failureCause | FMEA 테이블 | FMEA 전용 |
| **CP 전용** | detectorEp, detectorAuto, sampleSize, sampleFreq | CP 테이블 | CP 전용 |
| **PFD 전용** | productSC, processSC | PFD 테이블 | PFD 전용 |

### 구현 전략

```
Phase 1: 현재 구조 유지 + 동기화 강화
  → ControlPlanItem, PfdItem, PFMEA 별도 유지
  → 동기화 로직으로 연계성 확보
  
Phase 2: 종합 DB 마이그레이션 (향후)
  → UnifiedProcessItem 테이블 신설
  → 공통 필드는 종합 DB로 이관
  → 각 문서 테이블은 문서별 고유 필드만 유지
```

## 1️⃣ 원자성 데이터 저장 규칙

### 기본 원칙
```
Excel 1행 = DB 1레코드
병합 셀도 펼쳐서 개별 저장 (병합 정보는 화면에서 동적 계산)
```

### 원자성 저장 흐름
```
[Excel 병합 셀]
      ↓
[1단계: 병합 영역 값 추출]
      ↓
[2단계: 원자성 행 변환] → 각 Excel 행 = 1개 DB 레코드
      ↓
[3단계: 상속 컬럼 값 상속] → 병합 컬럼은 이전 행 값 상속
      ↓
[4단계: DB 저장] → 모든 필드가 원자적으로 저장
      ↓
[5단계: 화면 렌더링] → rowSpan 동적 계산으로 병합 표시
```

## 2️⃣ 모자관계 (Parent-Child) 구조

### 핵심 구조도
```
공정 (processNo + processName)
│
├─ 공정설명 (processDesc)
│   │
│   ├─ 부품명 (partName) ─────→ 제품특성 (productChar)
│   │                              └→ 특별특성 (specialChar)
│   │
│   └─ 설비 (equipment) ──────→ 공정특성 (processChar)
│                                  └→ 특별특성 (specialChar)
│
└─ [FMEA 전용] 작업요소 (workElement) → 4M+1E 분류
```

### 모자관계 정의표

| 부모 (Parent) | 자식 (Child) | 예시 | 관계 |
|---------------|--------------|------|------|
| **공정** | 공정설명 | 010절단 → 강관절단 | 1:N |
| **공정설명** | 부품명, 설비 | 강관절단 → 스틸파이프, UNCOILING | 1:N |
| **부품명** | 제품특성 | 스틸파이프 → 외경, 두께, 진원도 | 1:N |
| **설비** | 공정특성 | UNCOILING → RPM, 교환주기, 톱날흔들림 | 1:N |

### DB 저장 예시

```
행1: processNo=010, partName=스틸파이프, equipment=UNCOILING, productChar=외경, processChar=RPM
행2: processNo=010, partName=스틸파이프, equipment=UNCOILING, productChar=두께, processChar=교환주기
행3: processNo=010, partName=스틸파이프, equipment=UNCOILING, productChar=진원도, processChar=톱날흔들림
행4: processNo=010, partName=스틸파이프, equipment=WELDING, productChar=-, processChar=용접전류
```

### 제품특성 / 공정특성 상세 규칙

| 특성 | 부모 | 상속 | 병합 | 설명 |
|------|------|------|------|------|
| **제품특성** (productChar) | **partName** (부품명) | ❌ 상속 안 함 | ✅ 동일값 병합 | 부품의 제품 품질 특성 |
| **공정특성** (processChar) | **equipment** (설비) | ❌ 상속 안 함 | ❌ 병합 안 함 | 설비의 공정 관리 특성 |

#### 제품특성 (productChar)
- **부모**: `partName` (부품명)
- **상속**: ❌ 하지 않음 (각 행에 고유 값)
- **병합**: ✅ 동일 값이 연속되면 병합
- **예시**: 스틸파이프 → 외경, 두께, 진원도

#### 공정특성 (processChar)
- **부모**: `equipment` (설비)
- **상속**: ❌ 하지 않음 (각 행에 고유 값)
- **병합**: ❌ 하지 않음 (개별 표시)
- **예시**: UNCOILING 설비 → RPM, 교환주기, 톱날흔들림

#### 특별특성 (specialChar)
- **부모**: productChar 또는 processChar
- **상속**: ❌ 하지 않음
- **병합**: ❌ 하지 않음
- **예시**: CC, SC, IC

## 4️⃣ FMEA 4M+1E 작업요소 분류

| 분류코드 | 영문 | 한글 | 내용 | 예시 |
|---------|------|------|------|------|
| **MC** | Machine | 설비 | 설비, 금형, 지그 | PRESS, MOLD |
| **IM** | Indirect Material | 부자재 | 간접자재 | 용접봉, 윤활유 |
| **MN** | Man | 작업자 | 사람, 작업방법 | 숙련도, 교육 |
| **EN** | Environment | 환경 | 작업환경 | 온도, 습도 |

### CP/PFD vs FMEA 필드 차이

| 문서 | UI 컬럼명 | DB 필드 | 용도 |
|------|----------|---------|------|
| **CP/PFD** | 설비/금형/JIG | `equipment` | 설비 표시 + 공정특성 부모 |
| **FMEA** | 작업요소 | `workElement` | 4M+1E 분류 (MC/IM/MN/EN) |

**⚠️ 주의**: CP/PFD의 `equipment`와 FMEA의 `workElement`는 **별개 필드**

## 7️⃣ rowSpan 동적 계산 규칙

### 훅 목록

| 훅 | 기준 | 적용 컬럼 |
|----|------|----------|
| useProcessRowSpan | 연속된 같은 processNo+processName | 공정번호, 공정명 |
| useDescRowSpan | 연속된 같은 processNo+processName+processLevel+processDesc | 레벨, 공정설명 |
| usePartNameRowSpan | 연속된 같은 partName 값 | 부품명 |
| useWorkRowSpan | 연속된 같은 **equipment** 값 | 설비/금형 |
| useCharRowSpan | 연속된 같은 productChar 값 | 제품특성 |

### 병합 그룹 키 구성

```typescript
// 공정 그룹 키
processKey = `${processNo}-${processName}`;

// 공정설명 그룹 키
descKey = `${processNo}-${processName}-${processLevel}-${processDesc}`;

// 설비 그룹 키 (공정특성의 부모)
equipmentKey = `${processNo}-${processName}-${processLevel}-${processDesc}-${equipment}`;
```

## 🔟 FMEA 단계별 확정 → DB 저장

### FMEA Stage 흐름

```
Stage 1: 기본정보      → ProjectLinkage, FMEAHeader
    ↓
Stage 2: 구조분석 확정 → L1/L2/L3 Structure
    ↓                   ★ UnifiedProcessItem 동기화 (processNo, processName, workElement)
    ↓
Stage 3: 기능분석 확정 → L1/L2/L3 Function
    ↓                   ★ UnifiedProcessItem 업데이트 (productChar, processChar, specialChar)
    ↓
Stage 4: 고장연결 확정 → FE/FM/FC, FailureLink, FailureAnalysis
    ↓
Stage 5: 리스크분석    → RiskAnalysis (S/O/D/AP)
    ↓                   ★ CP에 refSeverity, refOccurrence, refDetection 연동
    ↓
Stage 6: 최적화        → Optimization
```

### 단계별 저장 테이블

| Stage | 확정 내용 | 저장 테이블 | UnifiedProcessItem 동기화 |
|-------|----------|------------|--------------------------|
| **2** | 구조분석 | L1/L2/L3 Structure | processNo, processName, workElement |
| **3** | 기능분석 | L1/L2/L3 Function | productChar, processChar, specialChar |
| **4** | 고장연결 | FE/FM/FC, FailureLink | - (FMEA 전용) |
| **5** | 리스크 | RiskAnalysis | - (CP ref 필드 연동) |
| **6** | 최적화 | Optimization | - (FMEA 전용) |

```

│                    CP 원자성 데이터 흐름                         │
```
