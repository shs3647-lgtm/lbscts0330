```
├── page.tsx                  ← 홈 → 가족 선택 화면
└── globals.css
```

### 2.2 각 가족의 전용 레이아웃

```tsx
// src/app/(fmea-core)/layout.tsx
import dynamic from 'next/dynamic';

// ★ 사이드바를 FMEA Core 메뉴만 표시하는 경량 버전으로
const FmeaSidebar = dynamic(() => import('@/components/layout/FmeaSidebar'), {
  ssr: false,
  loading: () => <div className="w-14 bg-gray-800" />,
});

export default function FmeaCoreLayout({ children }: { children: React.ReactNode }) {
  return (
    <div className="flex h-screen">
      <FmeaSidebar />
      <main className="flex-1 overflow-auto">{children}</main>
    </div>
  );
}
```

### 2.3 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **Code Splitting** | 가족1 진입 시 가족2/3 코드는 로드하지 않음 |
| **DB 공유** | 동일 PostgreSQL + 동일 Prisma 스키마 (모든 가족 공유) |
| **인증 공유** | 로그인/세션은 루트 레이아웃에서 처리 |
| **API 공유** | /api 라우트는 모든 가족이 공유 |
| **독립 장애** | 가족2에 버그가 있어도 가족1에 영향 없음 |

## 3. 마이그레이션 단계

### Phase 1: Route Group 생성 (1일)

**작업**: 기존 모듈을 Route Group 폴더로 이동

```bash
# 가족1: FMEA Core
mkdir -p "src/app/(fmea-core)"
mv src/app/pfmea    "src/app/(fmea-core)/pfmea"
mv src/app/dfmea    "src/app/(fmea-core)/dfmea"
mv src/app/control-plan "src/app/(fmea-core)/control-plan"
mv src/app/pfd      "src/app/(fmea-core)/pfd"
mv src/app/master   "src/app/(fmea-core)/master"

# 가족2: Planning
mkdir -p "src/app/(planning)"
mv src/app/apqp     "src/app/(planning)/apqp"
mv src/app/gantt    "src/app/(planning)/gantt"

# 가족3: Operations
mkdir -p "src/app/(operations)"
mv src/app/ws       "src/app/(operations)/ws"
mv src/app/pm       "src/app/(operations)/pm"

# 공통
mkdir -p "src/app/(common)"
mv src/app/login    "src/app/(common)/login"
mv src/app/auth     "src/app/(common)/auth"
mv src/app/admin    "src/app/(common)/admin"
mv src/app/dashboard "src/app/(common)/dashboard"
mv src/app/welcomeboard "src/app/(common)/welcomeboard"
```

**URL 영향**: Route Group `()` 괄호는 URL에 포함되지 않음
- `/pfmea/register` → 그대로 `/pfmea/register` (변경 없음)
- `/apqp` → 그대로 `/apqp` (변경 없음)

### Phase 2: 가족별 레이아웃 분리 (1일)

| 파일 | 역할 |
|------|------|
| `src/app/(fmea-core)/layout.tsx` | FMEA Core 사이드바 (PFMEA/DFMEA/CP/PFD 메뉴만) |
| `src/app/(planning)/layout.tsx` | Planning 사이드바 (APQP/Gantt 메뉴만) |
| `src/app/(operations)/layout.tsx` | Operations 사이드바 (WS/PM 메뉴만) |

**현재 Sidebar.tsx (707줄)** → 가족별로 3개의 경량 사이드바로 분리:
- `FmeaSidebar.tsx` (~200줄) — PFMEA/DFMEA/CP/PFD 메뉴만
- `PlanningSidebar.tsx` (~80줄) — APQP/Gantt 메뉴만
- `OperationsSidebar.tsx` (~80줄) — WS/PM 메뉴만

### Phase 4: 불필요 의존성 제거 (0.5일)

FMEA Core에서 불필요한 라이브러리를 lazy import로 전환:

```tsx
// recharts — dashboard에서만 사용 → (common)/dashboard에서 dynamic import
const Chart = dynamic(() => import('recharts').then(m => m.BarChart), { ssr: false });

// gantt-task-react — (planning)에서만 사용 → dynamic import
const Gantt = dynamic(() => import('gantt-task-react').then(m => m.Gantt), { ssr: false });

// react-pdf — approval에서만 사용 → dynamic import
const PDFViewer = dynamic(() => import('react-pdf'), { ssr: false });
```

## 6. 예상 효과

### 6.1 번들 크기 감소

| 시나리오 | 초기 로딩 번들 | 감소율 |
|----------|----------------|--------|
| 현재 (모놀리식) | 전체 앱 (~3MB+) | 기준 |
| Route Group 분리 후 (FMEA Core 진입) | FMEA Core만 (~1.5MB) | **~50%** |
| 불필요 라이브러리 lazy 처리 후 | FMEA Core (~1MB) | **~67%** |

### 6.2 로딩 시간 개선

| 환경 | 현재 | 개선 후 |
|------|------|---------|
| 로컬 네트워크 (100Mbps) | 3-5초 | 1-2초 |
| 저속 네트워크 (10Mbps) | 10-30초 (타임아웃 위험) | 3-5초 |
| 개발 서버 (npm run dev) | 5-10초 | 프로덕션 빌드로 대체 |

### 6.3 안정성 개선

- FMEA Core 시연 시 Planning/Operations 코드 미로드 → **메모리 50% 절감**
- 가족 간 상태 격리 → **모듈 간 간섭 원천 차단**
- 가족별 error.tsx → **다른 가족 에러가 FMEA Core에 영향 안 줌**
