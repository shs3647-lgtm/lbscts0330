### 2.1 관리유형 핵심 판별 기준

```
★ AIAG-VDA 핵심: O값은 "관리유형(Type of Control)"이 결정한다

  O=1:     Technical (설계/공정으로 물리적 방지)
            → 부품형상, 지그/치공구 설계로 고장원인 자체가 불가능
            → "실수가 물리적으로 발생할 수 없는" 수준

  O=2~3:   Best Practices (모범사례 — Behavioral 또는 Technical)
            → 입증된 방법으로 매우 효과적
            → SPC, Poka-Yoke 검증 완료, 연 1회 이하 발생

  O=4~5:   Behavioral or Technical (효과적)
            → 관리를 통해 효과적으로 예방
            → 연 1+ ~ 월 1+ 발생

  O=6~7:   Behavioral or Technical (다소 효과적)
            → 어느 정도 효과있지만 완전하지 않음
            → 주 1+ ~ 일 1+ 발생

  O=8~9:   Behavioral (행동적 — 거의 효과 없음)
            → 작업자 교육/표준서만 있고 기술적 장치 없음
            → 교대조당 1+ 발생

  O=10:    None (예방관리 없음)
```

### 4.2 알고리즘

def recommend_pc_1(fc_value):
    """추천1: FC→PC 직접매칭"""
    pc_set = fc_pc_map.get(fc_value, set())
    candidates = []

## 5. 추천2: FM→PC 교차매칭 + 관리유형 개선

### 5.1 원리

추천1에 **같은 FM의 다른 FC에서 사용하는 PC**를 합집합으로 추가.
**관리유형 등급으로 O값 적정성 평가** + **개선 방향 제시**.

```
추천2 = 추천1(FC→PC) ∪ { 같은 FM의 다른 FC에서 사용된 PC }

관리유형 개선 평가:
  현재PC의 적정O vs 추천2 PC의 적정O 비교
  → 추천2에 더 낮은 O의 PC가 있으면 "↓개선" (Technical → B+T 등)
  → 현재O < 적정O이면 "⚠ 과소평가" 경고
```

### 5.2 통계

| 항목 | 값 |
|------|-----|
| PC 2+ FM | 93개 / 107개 FM (87%) |
| 추천2 PC 추가 행 | **566건 (96%)** |
| 추가 PC 획득 FM | 93개 |

> DC 추천2의 22%보다 **훨씬 높은 96%** → FM→PC 교차가 PC에서 더 효과적

### 5.3 관리유형 개선 사례

**■ 사례1: 같은 FM, 다른 FC → 관리유형 업그레이드**

FM = "50번 커넥터 핀휨"

### 5.5 추천2 알고리즘

```python
PC_MATURITY = {
    # Behavioral 예시
    '검사원 셋업 표준 교육':        {'type': 'Behavioral',     'o_proven': 7, 'o_unproven': 9},
    '커넥터 취급 주의 교육':        {'type': 'Behavioral',     'o_proven': 7, 'o_unproven': 9},
    # Technical 예시
    'MC-검사기 Calibration 정기교정': {'type': 'Technical',      'o_proven': 4, 'o_unproven': 7},
    'MC-핀휨 검사지그 점검':        {'type': 'Technical',      'o_proven': 4, 'o_unproven': 7},
    # Behavioral+Technical 예시
    '셋업 표준 교육 및 셋업 검증':   {'type': 'Behavioral+Technical', 'o_proven': 4, 'o_unproven': 7},
}

def recommend_pc_2_with_control_type(fc_value, fm_value, current_pc, current_o):
    # 1. PC 후보 수집
    pc_set = set(fc_pc_map.get(fc_value, set()))   # 추천1
    pc_set |= set(fm_pc_map.get(fm_value, set()))   # 추천2 확장
    
    # 2. PC별 관리유형 부여
    candidates = []
    for pc in pc_set:
        mat = PC_MATURITY.get(pc, {'type': 'Behavioral', 'o_proven': 7})
        candidates.append({
            'pc': pc,
            'type': mat['type'],
            'o_proven': mat['o_proven'],
            'o_unproven': mat['o_unproven'],
        })
    
    # 3. 현재 O값 적정성 평가
    cur_mat = PC_MATURITY.get(current_pc, {'o_proven': 7})
    o_gap = current_o - cur_mat['o_proven']  # 음수 = 과소평가
    
    # 4. 최적 PC (가장 낮은 O)
    best = min(candidates, key=lambda x: x['o_proven'])
    
    return {
        'candidates': sorted(candidates, key=lambda x: x['o_proven']),
        'o_gap': o_gap,
        'o_warning': o_gap < 0,   # True면 과소평가
        'best_pc': best['pc'],
        'best_o': best['o_proven'],
        'improvement': cur_mat['o_proven'] - best['o_proven'],  # 양수면 개선
    }
```
