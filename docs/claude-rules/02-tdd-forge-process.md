# Rule 1: TDD 필수 + Forge 6단계 개발 프로세스

> **모든 코드 수정은 반드시 테스트 먼저 수정/작성한 후 구현 코드를 수정한다.**

---

## TDD 4단계 검증 체계

| 단계 | 명칭 | 방법 | 용도 |
|------|------|------|------|
| **0단계** | **테스트 먼저 (TDD)** | 기대 동작을 테스트로 작성 → 실패 확인 → 보고 | 요구사항을 코드로 정의 |
| 1단계 | **타입 체크** | `npx tsc --noEmit` | 타입 에러 0개 |
| 2단계 | **단위 테스트** | Vitest (structureGuard, 비정상 엑셀 47개) | 함수별 정확성 |
| 3단계 | **통합 테스트 (E2E)** | Playwright 브라우저 검증 | 전체 흐름 정상 |

### TDD 절차 (0단계 상세)

```
1. 기대 동작을 테스트 코드로 **먼저** 작성/수정
2. 테스트 실행 → FAIL 확인 (Red)
3. 사용자에게 "이 테스트가 실패합니다. 이제 구현을 수정합니다." 보고
4. 구현 코드 수정
5. 테스트 실행 → PASS 확인 (Green)
```

- ❌ **구현 먼저 수정 → 테스트 맞춰 변경 = TDD 위반 (금지)**
- ✅ **테스트 먼저 수정 → 실패 확인 → 구현 수정 → 통과 = 올바른 TDD**

### 검증 단계 누적 원칙

- 1단계(tsc)만 통과 → "완료" 금지 (타입만 확인)
- 2단계로 **각 함수가 올바르게 동작하는지** 검증
- 3단계로 **실제 사용 시나리오** 확인
- **7단계(구조→기능→고장→위험→최적화→CP→PFD) E2E 검증 통과 후 다음 단계**

## Forge 6단계 프로세스 (비-트리비얼 작업 필수)

> 3파일+ 변경, UI 전체 수정, 아키텍처 변경 시 필수 적용.
> 단순 타이포/1-2줄 수정은 예외 (Forge 생략 가능).

| 단계 | 명칭 | 방법 | 산출물 |
|------|------|------|--------|
| 1 | **EXPLORE** | 코드베이스 탐색, 현재 상태 파악 | 현황 분석 보고 |
| 2 | **PLAN** | 수정 파일 목록 + 변경 내용 + 검증 기준 | 승인된 계획 |
| 3 | **TDD** | Playwright/Vitest 테스트 먼저 작성 → RED | 실패하는 테스트 |
| 4 | **EXECUTE** | 계획대로만 구현 (계획 외 변경 금지) | 구현 코드 |
| 5 | **VERIFY** | `tsc --noEmit` + 테스트 PASS + Playwright | 검증 보고서 |
| 6 | **COMMIT** | 검증 통과 후 커밋 | 커밋 해시 |

### VERIFY-LOOP

```
VERIFY 실패 → EXECUTE 돌아가 수정 → 재검증 (루프)
Playwright 브라우저 검증은 UI 변경 시 필수
```

### PRD 학습 (훅 시작 전)

> Forge EXPLORE 단계에서 관련 PRD 문서(`docs/prd/`)를 반드시 읽고 요구사항 파악.
> PRD가 없으면 사용자에게 요구사항 정의 요청.

## 에이전트 독립 임무 + 교차 검증

### 원칙

- 각 에이전트는 **독립적인 임무** 수행 (탐색/계획/구현/검증 분리)
- 구현 에이전트와 검증 에이전트는 **다른 에이전트**가 담당
- 최종 검증은 반드시 **Playwright E2E 테스트**

### 에이전트 역할 분리

| 역할 | 임무 | 도구 |
|------|------|------|
| **탐색 에이전트** | 코드베이스 분석, 영향 범위 파악 | Explore agent |
| **계획 에이전트** | PRD 학습, 수정 계획 수립 | Plan agent |
| **구현 에이전트** | 계획에 따라 코드 작성 | gsd-executor |
| **검증 에이전트** | 구현 결과 독립 검증 | gsd-verifier |
| **E2E 에이전트** | Playwright 브라우저 테스트 | Playwright |

### 교차 검증 흐름

```
구현 에이전트 완료 → 검증 에이전트 투입
  ├── tsc --noEmit 통과?
  ├── 단위 테스트 PASS?
  ├── 파이프라인 검증 allGreen?
  ├── Playwright E2E PASS?
  └── 브라우저 렌더링 성공? → 커밋 허용
```

## 파이프라인 보호 — 방어 코드 + Auto-Fix + 루프

### 코드 성공 후 방어 체계

```
코드 수정 완료
  → tsc --noEmit (타입 방어)
  → pipeline-verify POST (5단계 검증)
    → FAIL → auto-fix 루프 (최대 3회)
    → PASS → 커밋 허용
  → rebuild-atomic (DC/PC 최신화)
  → export-master (마스터 동기화)
  → import-validation (16개 규칙)
```

### 훅 검증 Audit-Fix 루프

```
구축 완료 → 훅 검증 시작
  → Guard 테스트 실행 (FK/Import/카테시안)
  → 실패 시 자동 수정 시도
  → 재검증 (최대 3회)
  → 전체 PASS → 코드프리즈 적용
```

### 실패 시 반복 루프

```
검증 실패 → 원인 분석 → 수정 → 재검증
  ↑___________________________|  (최대 3회)
  3회 실패 → 사용자에게 보고 + 수동 개입 요청
```
