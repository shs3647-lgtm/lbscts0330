# 구조드리븐(Structure-Driven) 설계 — PFMEA 자동매핑

> **핵심 원칙**: 구조(호텔)를 먼저 콘크리트로 짓고, 데이터(손님)를 검증 후 입장시킨다.
> 구조는 절대 불변. 데이터가 잘못되면 데이터를 거부한다 (구조를 변경하지 않는다).

---

## 1. 전체 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│                    구조분석 (Structure)                    │
│   사용자가 구조분석 탭에서 직접 설정 (절대 자동 변경 안 됨)     │
│                                                         │
│   L1: 완제품명 (차량용 서브프레임)                          │
│   L2: 공정 목록 (10.절단, 20.성형, 30.용접 ...)            │
│   L3: 작업요소 (10-MC 절단기, 10-MN 작업자, 30-IM 소모품)  │
└───────────────┬─────────────────────────────────────────┘
                │ 구조 확정 (콘크리트)
                ▼
┌─────────────────────────────────────────────────────────┐
│              문지기 (Gatekeeper) 검증                      │
│                                                         │
│   1. 호텔 방 목록 생성 (RoomLock[])                       │
│      - L2탭: processNo + itemCode                       │
│      - L3탭: processNo + m4 + itemCode                  │
│                                                         │
│   2. 마스터 데이터(키) 검증                                │
│      - 방 존재 확인 (processNo, m4)                      │
│      - itemCode 유효성                                   │
│      - value 비어있지 않은지                               │
│      - sourceFmeaId 일치                                 │
│                                                         │
│   3. 연쇄 삭제 (Cascade Rejection)                       │
│      - 비정상 데이터 발견 → 같은 그룹 전체 삭제              │
│      - [데이타 누락] 표시 → 사용자 수동 수정                │
└───────────────┬─────────────────────────────────────────┘
                │ 검증 통과 데이터만
                ▼
┌─────────────────────────────────────────────────────────┐
│            트리뷰 미리보기 (Preview Modal)                  │
│                                                         │
│   사용자가 매핑 결과를 확인하고 승인/취소                     │
│   - 정상 매칭: 녹색 표시                                   │
│   - 데이터 누락: 적색 [데이타 누락] 배지                    │
│   - 거부된 항목: 주황색 거부 목록                           │
└───────────────┬─────────────────────────────────────────┘
                │ 사용자 승인
                ▼
┌─────────────────────────────────────────────────────────┐
│           구조 보호 매핑 (Structure Guard)                  │
│                                                         │
│   1. 기존 구조(L2/L3) 유지                                │
│   2. functions[]/processChars[]에 행추가(append)만 허용     │
│   3. 매핑 후 스냅샷 비교 (변경 전 vs 후)                    │
│   4. 구조 위반 감지 시 → 원본 state 강제 복원               │
│   5. 빈 functions 자동 보정 (최소 1행 보장)                 │
└─────────────────────────────────────────────────────────┘
```

## 2. 계층별 구조 보호

### 2.1 L1 (완제품) — 구분(YP/SP/USER) + 제품기능 + 요구사항

```
L1 완제품: "차량용 서브프레임"
├── YP (후공정)
│   ├── C2 제품기능: "치수 정밀도를 유지한다"    ← 자동매핑 대상
│   └── C3 요구사항: "치수 공차"               ← 자동매핑 대상
├── SP (Ship to Plant, 상위고객)
│   ├── C2 제품기능: ...
│   └── C3 요구사항: ...
└── USER (최종사용자)
    ├── C2 제품기능: ...
    └── C3 요구사항: ...
```

| 필드 | 보호 수준 | 설명 |
|------|----------|------|
| `l1.name` | **IMMUTABLE** | 완제품명 — 변경 불가 |
| `l1.types[].name` | **IMMUTABLE** | 구분(YP/SP/USER) — 변경 불가 |
| `l1.types[].functions[]` | APPENDABLE | 제품기능 — 행추가만 |
| `l1.types[].functions[].requirements[]` | APPENDABLE | 요구사항 — 행추가만 |

**Gatekeeper 검증**: C1(구분)이 YP/SP/USER에 해당하는지 확인 → ETC 등 비표준 구분은 `NO_ROOM` 거부

---

### 2.2 L2 (공정) — 공정기능 + 제품특성

```
L2 공정 목록 (구조분석에서 확정, 공정코드 옵션 포함):
├── 10. [A-10] 절단(컷팅)
│   ├── A3 공정기능: "원자재를 도면 지정 치수로 절단한다"  ← 자동매핑
│   └── A4 제품특성: "외경/두께", "길이"                ← 자동매핑
├── 20. [A-20] 성형/가공
│   ├── A3 공정기능: ...
│   └── A4 제품특성: ...
├── 30. [A-30] 용접/접합
├── 40. [B-10] 열처리
├── 50. [B-20] 표면처리/도장
├── 60. [C-10] 조립
└── 70. [C-20] 검사
```

| 필드 | 보호 수준 | 설명 |
|------|----------|------|
| `l2[].id` | **IMMUTABLE** | 공정 ID — 삭제/교체 불가 |
| `l2[].no` | **IMMUTABLE** | 공정번호 — 변경 불가 |
| `l2[].code` | **OPTIONAL** | 공정코드 — 사용자 선택 입력 (예: A-10, B-20) |
| `l2[].name` | **IMMUTABLE** | 공정명 — 변경 불가 |
| `l2[].functions[]` | APPENDABLE | 공정기능 — 행추가만 |
| `l2[].functions[].productChars[]` | APPENDABLE | 제품특성 — 행추가만 |

**Gatekeeper 검증**: 마스터 데이터의 `processNo`가 `l2[].no`에 존재하는지 확인 → `processNo=80` 등 없는 공정은 `NO_ROOM` 거부

---

### 2.3 L3 (작업요소) — 요소기능 + 공정특성

```
 L3: MN 작업자
│   ├── B2 요소기능: ...
│   └── B3 공정특성: .
├── L3: MC 절단기(밴드소/레이저/워터젯)
│   ├── B2 요소기능: "원자재를 지정 치수로 절단한다"  ← 자동매핑
│   └── B3 공정특성: "RPM/이송속도", "절삭유 유량"   ← 자동매핑
├── L3: MC 절삭공구(톱날/비트)
│   ├── B2 요소기능: ...
│   └── B3 공정특성: ...
├── L3: MN 작업자 (CP·PFD·PM 연동 추가)
│   ├── B2 요소기능: ...
│   └── B3 공정특성: ...
├── L3: IM 부자재 (원자재 제외, CP·PFD·PM 연동)
│   ├── B2 요소기능: ...
│   └── B3 공정특성: ...
└── L3: EN 작업장 온도/습도/조도/환기 환경
    ├── B2 요소기능: ...
    └── B3 공정특성: ...
```

| 필드 | 보호 수준 | 설명 |
|------|----------|------|
| `l3[].id` | **IMMUTABLE** | 작업요소 ID |
| `l3[].m4` | **IMMUTABLE** | 4M 분류(MN/MC/IM/EN) — 변경 불가 |
| `l3[].name` | **IMMUTABLE** | 작업요소명 — 변경 불가 |
| `l3[].functions[]` | APPENDABLE | 요소기능 — 행추가만 |
| `l3[].functions[].processChars[]` | APPENDABLE | 공정특성 — 행추가만 |

**Gatekeeper 검증**: `processNo + m4` 복합키로 방 매칭
- `processNo=10, m4=MC` → `10::MC` 방 존재 → 통과
- `processNo=10, m4=IM` → `10::IM` 방 없음 → `WRONG_ROOM` 거부
- `processNo=99, m4=MC` → 공정 99 없음 → `NO_ROOM` 거부

### 2.4 고장분석 — L1(고장영향) + L2(고장형태) + L3(고장원인)

```
L1 고장영향 (C4):
```

## 3. 문지기(Gatekeeper) 검증 흐름

### 3.1 호텔 비유

| 개념 | 비유 | 실제 |
|------|------|------|
| 호텔 | 구조 | 구조분석에서 확정된 L2/L3 계층 |
| 방 | 셀 | processNo + m4 + itemCode |
| 자물쇠 | 매칭 기준 | RoomLock (기대하는 데이터 종류) |
| 키 | 마스터 데이터 | DataKey (processNo + m4 + itemCode + value) |
| 문지기 | 검증 로직 | validateAutoMapping() |

### 3.2 검증 순서

```
마스터 데이터 아이템 하나씩:

1. sourceFmeaId 확인 → 불일치 시 HOTEL_MISMATCH
2. itemCode 유효성 → 허용 목록에 없으면 WRONG_KEY_TYPE
3. value 비어있음 → EMPTY_KEY
4. 방 존재 확인 (processNo + m4) → 없으면 NO_ROOM / WRONG_ROOM
5. 공통공정(0/00) 와일드카드 → 같은 m4의 모든 방에 분배
6. ✅ 통과 → matched 목록에 추가
```

### 3.3 연쇄 삭제 (Cascade Rejection)

비정상 데이터가 발견되면, 같은 `processNo+m4` 그룹의 **모든** 데이터를 거부합니다.

```
예시: processNo=20, m4=MC 그룹

  B6 "잘못된 코드" → WRONG_KEY_TYPE (트리거!)
  ↓ 연쇄 삭제
  B2 "정상 기능"   → CASCADE (같은 그룹이므로 연쇄 삭제)
  B3 "정상 특성"   → CASCADE

→ 20::MC 전체가 [데이타 누락]으로 렌더링
→ 사용자가 수동으로 올바른 데이터 입력
```

**연쇄 삭제 트리거 사유**:
- `NO_ROOM`: 구조에 없는 공정번호
- `WRONG_ROOM`: m4 불일치/누락
- `WRONG_KEY_TYPE`: itemCode 불일치
- `HOTEL_MISMATCH`: FMEA ID 불일치

**연쇄 삭제 비트리거** (개별 항목만 거부):
- `EMPTY_KEY`: 빈 값 — 단순 누락이므로 같은 그룹의 정상 데이터는 유지

## 4. 구조 보호 가드 (Structure Guard)

### 4.1 스냅샷 비교 방식

```typescript
// 매핑 전: 원본 state 스냅샷 촬영
const before = takeStructureSnapshot(originalState);

// 매핑 로직 실행 (functions[] 행추가)
const newState = applyAutoMapping(originalState, matchedData);

// 매핑 후: 변경된 state 스냅샷 촬영
const after = takeStructureSnapshot(newState);

// 비교: 구조 필드가 변경되었는지 확인
const violations = validateStructuralIntegrity(before, after);

if (violations.length > 0) {
  // ❌ 구조 위반! → 원본 state 강제 복원
  return originalState;
}

// ✅ 구조 정상 → 빈 함수 보정 후 반환
return ensureMinimumFunctions(newState);
```

### 4.2 감시 대상

| 위반 유형 | 설명 | 대응 |
|----------|------|------|
| L2_DELETED | L2 공정 삭제됨 | 원본 복원 |
| L2_MODIFIED | L2 공정번호 변경됨 | 원본 복원 |
| L3_DELETED | L3 작업요소 삭제됨 | 원본 복원 |
| L3_MODIFIED | L3의 m4 변경됨 | 원본 복원 |

### 4.3 최소 함수 보장

매핑 후 L3의 `functions[]`가 빈 배열이면 placeholder 자동 삽입:

```
functions = [] → functions = [{ name: '', processChars: [{ name: '', specialChar: '' }] }]
```

이렇게 하면 워크시트 렌더링 시 빈 행이 하나 표시되어, 사용자가 수동으로 입력할 수 있습니다.

```
→ 결과: [유효 데이터만 매핑] + 거부 항목 수동 수정
```

| 데이터 상태 | 처리 | 사용자 액션 |
