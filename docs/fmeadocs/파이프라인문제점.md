## 2. 코드 근거 (저장·파서)

- `validFLs = failureLinks.filter(fl => fl.fmId && fl.feId && fl.fcId)`

`src/lib/fmea/cross-sheet-resolver.ts` — `resolveFE` / `resolveFM` / `resolveFC`  
`position-parser.ts` — FC 행과 L3 인덱스 1:1 가정 등.

### 3.6 Import 화면 통계 vs PostgreSQL vs 워크시트 (추가 진단, 2026-03-28)

> **가설(사용자 제기):** Import 화면 숫자는 “문제 없어 보이는데”, 실제 PG에는 일부가 안 들어갔을 수 있음 → **통계가 저장 실패를 가릴 수 있다.**  
> 아래는 코드·훅 기준 **3자 대조**다. (멀티 에이전트 탐색 + 핵심 파일 확인)

#### A. Import UI — 무엇을 세는가 (대부분 메모리)

| 영역 | 파일 | 데이터 원천 |
|------|------|-------------|
| 상단 카드 (공정·데이터 건·고장사슬) | `pfmea/import/legacy/page.tsx` | `flatData`, `masterChains` 등 **클라이언트 메모리** |
| `ImportStepBar` | `import/components/ImportStepBar.tsx` | `flatData.length`, `failureChains.length` **메모리** |
| `DataStatusBar` | `import/components/TemplateSharedUI.tsx` | `flatData` 파생 **메모리** |
| 통계표·L1/L2/L3 탭 행 수 | `import/components/TemplatePreviewContent.tsx` | `crossTab` / `effectiveStatistics` → **`flatData`·파생 체인** 중심 |
| **pgsql / API 열** | `useImportVerification.ts` | 사용자가 실행 시: `GET /api/fmea/verify-counts`, `GET /api/fmea?…&format=atomic` → **실제 PG/응답 기반** |

**중요:** 위치기반 Import에서 `parseResult?.statistics` → `TemplateGeneratorPanel` 경로는 **`useImportFileHandlers`가 `setParseResult`를 채우지 않으면** `parseStatistics`가 비어 있고, 표는 **`flatData` 기반 `effectiveStatistics`**에 의존할 수 있음 → **“파서 공식 통계”와 화면이 어긋날 여지**.

#### B. 저장 API — `success`와 실제 반영

| 항목 | 내용 |
|------|------|
| UI 저장 | `useImportFileHandlers.ts` → `POST /api/fmea/save-position-import` |
| 응답 `atomicCounts` | 트랜잭션 후 Prisma **`count()`** → **현재 스키마에 존재하는 총량**(전량 삭제 후 재삽입이면 ≈방금 반영분) |
| 응답 `stats` | 요청 payload의 **`atomicData.stats` (파서)** — DB 쓰기 결과와 **자동 정합되지 않음** |
| **FailureLink** | `fmId && feId && fcId` **삼중 FK**만 insert; 불완전 링크는 **로그 위주**, 응답에 **`skippedIncompleteFailureLinks` 같은 필드 없음** (★1과 연계) |
| **RiskAnalysis** | 유효 FL의 `linkId`에 한해 살림 → 스킵된 FL에 묶인 RA는 **DB에 안 감** |
| 기타 | `skipDuplicates`, `safeTx` no-op, `manualMode` 분기 등으로 **커밋 성공 ≠ 엑셀의 모든 행이 PG에 1:1 존재** 가능 |

**참고:** `raw-to-atomic`은 FL 필터가 **`fmId`만** 요구하는 구간이 있어 **`save-position-import`와 동일 시트라도 FL 건수가 달라질 수 있음** — 문서·테스트는 `save-from-import` 경로를 가리키는 경우가 있으나, **본 워크스페이스 `src/app/api/fmea`에는 `save-from-import` 라우트가 없음**(E2E/시드·문서와 **경로 드리프트** 가능) → 통합 Import는 **`save-position-import`를 SSoT로 명시**할 것.

#### C. 워크시트 L1/L2/L3·고장연결 — PG와 숫자가 어긋날 수 있는 이유

| 원인 | 설명 |
|------|------|
| `useWorksheetDataLoader` | L3가 비어 있으면 **placeholder L3** 주입 → 화면 행 수 > `l3_structures` 가능(저장 전) |
| L1 비어 있을 때 | `atomicToLegacy` 후 **YP/SP/USER 스캐폴드** — UI만, DB `l1_functions`와 불일치 가능 |
| `useRowsCalculation` | 병합·flat row → **표시 행 ≠ 엔티티 건수** |
| `FailureLinkTab` / `useLinkData` | 필터·미확정·의미 없는 이름 스킵 → **다이어그램/패널 건수 < `failure_links`** 가능 |
| 소프트 삭제 | API는 `deletedAt: null` 링크만 — raw `COUNT(*)`와 앱 불일치 |

#### D. “100% 일치”를 무엇으로 정의할지 (권장)

1. **엔티티 건수:** `verify-counts` / `GET /api/fmea` 기준 L2·L3·FM·FE·FC·FL·RA (동일 `fmeaId`, 프로젝트 스키마).  
2. **Import 직후:** 저장 응답 `atomicCounts` vs (선택) payload 내 파서 산출 **동일 정의**의 기대치 — **불일치 시 경고 배너**(이름 매칭·레거시 폴백 없이 **건수·코드별**만).  
3. **워크시트:** 구조 탭은 **원시 L1/L2/L3 엔티티 수**와 비교; 고장연결은 **`savedLinks`(DB GET) 건수**와 `failure_links` count 비교(필터 제외한 “DB 모드” 필요 시 별도 표시).

## 4. 권장 후속 조치

1. **교차 공정 42건** — `validate-fk` 상세의 `FL … FM→L2=…, FC→L2=…` 를 기준으로 FC 시트 **공정번호·L3 원본행** 재점검.  
2. **FL 없는 FM 2건** — 엑셀 FC 고장사슬에 해당 FM 행 존재 여부, `CrossSheetResolver` 해석 실패 로그(`POSITION_PARSER_VERBOSE=1`).  
3. **FL 없는 FC 11건** — FC는 존재하나 어떤 FL에도 `fcId`로 등장하지 않음 → 시트 체인 누락 또는 resolve 실패.  
4. **고장매칭 자동 버튼** — 카테시안 위험이 있으므로(내부 규정 참고), **엑셀·파서 정합을 우선**하고 자동은 보조로만 사용.  
5. **Import 통계 vs PG** — §3.6 및 §7 **★9~★13** 순서로 “화면이 거짓말하지 않게” 한 뒤, 기존 ★1~★8(파서·FL·교차공정) 개선.

### ★9 — Import 직후 “파서/메모리 기대 vs PostgreSQL” 자동 대조

| 항목 | 내용 |
|------|------|
| **문제** | 화면은 `flatData`·체인 길이로 “정상”처럼 보이나, FL/RA 필터·`skipDuplicates` 등으로 **PG 건수가 더 적을 수 있음** — 사용자가 pgsql 검증 버튼을 누르기 전까지 눈에 안 띔. |
| **목표** | `save-position-import` 성공 직후(또는 자가개선 루프 내) **동일 fmeaId**에 대해 `verify-counts` 또는 응답 `atomicCounts`와 **합의된 코드맵**(A1→L2 등)으로 1회 이상 비교, **불일치 시 배너/이슈 행**. |
| **실행 계획** | ① `mergeImportExpectedCounts` / `mapCountsToPgsql`와 동일 정의로 서버 또는 클라이언트에서 diff. ② **이름 매칭·레거시 blob 금지** — 건수·코드 단위만. |
| **완료 기준** | Import 완료 화면에서 “PG와 불일치 N건”이 0이거나, 불일치 시 원인 코드(FL 스킵 등)가 표시됨. |

## 8. 협업 방식에 대한 의견

- **★ 단위 + 표 업데이트**는 추적하기 좋다. **통계 거짓말(§3.6) 차단이 먼저**이므로 권장 순: **★9 → ★10 → ★11 → ★12** → 그다음 **★1 → ★2 → ★5 → ★3·★4** → P1(★6·★7·★8·★13).
- **Forge:** `tsc` → `npm run verify:all` → Playwright(`npm run test:e2e:final` / 필요 시 headed) → 만족 시 커밋; 실패 시 VERIFY–EXECUTE 루프.
- **완료할 때마다** 이 파일만 고쳐도 되고, 커밋 메시지에 `docs: 파이프라인문제점 ★N 완료` 를 붙이면 이력이 남는다.
- **코드프리즈 해제** 후에도 `save-position-import` / `position-parser` / `raw-to-atomic` 는 과거 사고 이력이 있으니, 변경은 **★별 PR** + `pipeline-verify` + `validate-fk` + `diag:pipeline-fk` 를 세트로 돌리는 것을 권장한다.
