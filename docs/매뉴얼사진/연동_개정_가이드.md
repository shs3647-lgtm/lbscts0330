## 1. ID 번호 규칙 (r00 개정 접미사)

### 1.1 기본 구조

모든 FMEA, CP(관리계획서), PFD(공정흐름도) 문서는 고유한 ID를 가집니다. ID 끝에는 개정 번호(`-r00`)가 자동으로 포함되어, 현재 문서가 몇 번째 개정인지 바로 알 수 있습니다.

우체국 등기번호에 배달 차수가 포함된 것과 같습니다. `-r00`은 "원본(초판)", `-r01`은 "1차 개정", `-r02`는 "2차 개정"을 의미합니다.

### 1.2 문서별 ID 형식

| 문서 | ID 형식 | 예시 | 설명 |
|------|---------|------|------|
| **PFMEA** | `pfm{YY}-{유형}{번호}-{연동}-r{NN}` | `pfm26-p001-s-r00` | 26년, Production, 001번, 단독, 초판 |
| **DFMEA** | `dfm{YY}-{유형}{번호}-{연동}-r{NN}` | `dfm26-m001-i01-r00` | 26년, Master, 001번, 연동그룹01, 초판 |
| **CP** | `cp{YY}-{유형}{번호}-{연동}-r{NN}` | `cp26-p001-s-r00` | 26년, Production, 001번, 단독, 초판 |
| **PFD** | `pfd{YY}-{유형}{번호}-{순번}-r{NN}` | `pfd26-p001-01-r00` | 26년, Production, 001번, 순번01, 초판 |

### 1.3 ID 구성 요소 상세

```
pfm26-p001-s-r00
 │  │  │  │ │ └── 개정번호: 00=초판, 01=1차개정, 02=2차개정...
 │  │  │  │ └──── 연동상태: s=단독(Solo), i01=연동그룹01
 │  │  │  └────── 일련번호: 001부터 자동 증가
 │  │  └───────── 유형: p=Production, m=Master, f=Family
 │  └──────────── 연도: 26=2026년
 └─────────────── 모듈: pfm=PFMEA, dfm=DFMEA, cp=CP, pfd=PFD
```

### 1.4 유형 코드

**FMEA/PFD 유형**

| 코드 | 의미 | 설명 |
|------|------|------|
| `m` | Master | 마스터 FMEA (기본 원본) |
| `f` | Family | 패밀리 FMEA (마스터 기반 파생) |
| `p` | Part | 부품별 FMEA |

**CP 유형**

| 코드 | 의미 | 설명 |
|------|------|------|
| `t` | Prototype | 시작품 단계 |
| `l` | Pre-Launch | 양산 전 단계 |
| `p` | Production | 양산 단계 |
| `s` | Safe Launch | 안전 런치 단계 |

### 1.5 연동 상태 코드

| 코드 | 의미 | 설명 |
|------|------|------|
| `s` | Solo (단독) | 다른 문서와 연동되지 않은 독립 문서 |
| `i01` | Linked (연동) | 연동 그룹 01에 소속된 문서 (01~99까지 가능) |

> **참고**: 과거 버전에서는 `L01` 형식을 사용했으나, 소문자 `l`과 숫자 `1`의 혼동을 방지하기 위해 현재는 `i01` 형식으로 통일되었습니다.

### 1.6 개정 번호 증가 규칙

```
초판 등록:        pfm26-p001-s-r00
1차 개정 (REV 1): pfm26-p001-s-r01
2차 개정 (REV 2): pfm26-p001-s-r02
  ...
N차 개정:        pfm26-p001-s-r{NN}
```

- 개정 시작은 개정관리 화면에서 "REV N+1" 버튼을 클릭하여 시작합니다
- 개정 번호는 자동으로 1씩 증가합니다 (수동 변경 불가)
- 연동된 CP/PFD 문서도 동일한 개정 번호를 공유하여, 세 문서의 버전이 항상 일치합니다

### 1.7 연동 문서 간 ID 대응

같은 프로젝트의 FMEA, CP, PFD는 ID의 중간 부분이 동일하므로 쉽게 대응 관계를 파악할 수 있습니다.

| PFMEA ID | CP ID | PFD ID |
|----------|-------|--------|
| `pfm26-p001-s-r00` | `cp26-p001-s-r00` | `pfd26-p001-01-r00` |
| `pfm26-m001-i01-r00` | `cp26-m001-i01-r00` | `pfd26-m001-01-r00` |

## 2. 6방향 연동 (CP↔PFD↔PFMEA)

### 2.1 연동 개요

PFMEA, CP, PFD 세 문서는 6개 방향으로 데이터를 주고받을 수 있습니다. 마치 삼각형의 세 꼭짓점을 양방향 화살표로 연결한 것과 같습니다. 어떤 문서에서 시작하더라도 나머지 두 문서로 데이터를 보낼 수 있습니다.

```
         PFMEA
        ↗     ↘
      ↙         ↖
   PFD ←────→ CP
```

### 2.2 연동 방향별 상세

#### (1) CP → PFD 연동

| 항목 | 내용 |
|------|------|
| **API** | `/api/pfd/sync-from-cp` |
| **방향** | 관리계획서 데이터를 공정흐름도로 전달 |
| **전달 데이터** | 공정번호, 공정명, 공정설명, 부품명(작업요소), 설비 |
| **병합 처리** | 같은 값을 가진 연속 행은 셀병합 상태를 유지합니다 |
| **기존 데이터** | 기존 PFD가 있으면 업데이트, 없으면 새로 생성합니다 |

#### (2) PFD → CP 연동

| 항목 | 내용 |
|------|------|
| **API** | `/api/control-plan/sync-from-pfd` |
| **방향** | 공정흐름도 데이터를 관리계획서로 전달 |
| **전달 데이터** | 공정번호, 공정명, 공정설명, 작업요소, 설비 |
| **기존 데이터** | 기존 CP가 있으면 업데이트, 없으면 새로 생성합니다 |

#### (3) CP → PFMEA 연동

| 항목 | 내용 |
|------|------|
| **API** | `/api/pfmea/sync-from-cp` |
| **방향** | 관리계획서 데이터를 PFMEA 구조분석으로 전달 |
| **1L 연동** | 완제품공정명(품명+생산공정)만 연동. 구분(YP/SP/EU), 기능, 요구사항은 FMEA에서 직접 입력 |
| **2L/3L 연동** | 공정번호, 공정명, 설비, 제품특성, 공정특성을 구조분석 L2/L3로 변환 |
| **기존 데이터** | 기존 FMEA가 있으면 업데이트, 없으면 새로 생성합니다 |

#### (4) PFMEA → CP 연동

| 항목 | 내용 |
|------|------|
| **상태** | CP에서 PFD를 거쳐 간접 연동 (PFMEA → PFD → CP 경로 사용) |
| **설명** | PFMEA 데이터를 먼저 PFD로 연동한 후, PFD에서 CP로 연동하는 2단계 경로입니다 |

#### (5) PFD → PFMEA 연동

| 항목 | 내용 |
|------|------|
| **API** | `/api/pfmea/sync-from-pfd` |
| **방향** | 공정흐름도 데이터를 PFMEA 구조분석으로 전달 |
| **1L 연동** | 완제품공정명(품명+생산공정)만 연동. 구분, 기능, 요구사항은 FMEA에서 직접 입력 |
| **2L/3L 연동** | 공정번호, 공정명, 공정설명, 작업요소를 구조분석 L2/L3로 변환 |
| **L3 매핑** | 설비 = equipment, 4M = MC(Machine) 고정 |
| **저장 방식** | DB에 직접 저장 (영구 보관) |

#### (6) PFMEA → PFD 연동

| 항목 | 내용 |
|------|------|
| **API** | `/api/pfd/sync-from-fmea` |
| **방향** | PFMEA 워크시트 데이터를 공정흐름도로 전달 |
| **전달 데이터** | L2 공정(공정번호, 공정명), L2 기능(제품특성), L3 작업요소(부품명, 설비), L3 기능(공정특성) |
| **특별특성 변환** | FMEA의 특별특성(C/CC/S/SC/I/IC) → PFD의 CC/SC/IC로 자동 변환 |

### 2.3 연동 데이터 매핑표

각 연동 방향에서 어떤 데이터가 어디로 전달되는지 정리한 표입니다.

| 출발 문서 필드 | → | 도착 문서 필드 | CP→PFD | PFD→CP | CP→FMEA | PFD→FMEA | FMEA→PFD |
|---------------|---|---------------|--------|--------|---------|----------|----------|
| 공정번호 | → | 공정번호 | O | O | O | O | O |
| 공정명 | → | 공정명 | O | O | O | O | O |
| 공정설명 | → | 공정설명 | O | O | - | O | - |
| 작업요소/부품명 | → | 작업요소/부품명 | O | O | O | O | O |
| 설비 | → | 설비 | O | O | O | O | O |
| 제품특성 | → | 제품특성 | - | - | O | - | O |
| 공정특성 | → | 공정특성 | - | - | O | - | O |
| 특별특성(CC/SC) | → | 특별특성(CC/SC) | - | - | - | - | O |

### 2.4 연동 실행 방법

> 📸 **[화면 캡처]** 연동 버튼 위치
>
> ![연동 버튼](화면캡처/31_연동버튼.png)

1. 연동을 **보내는** 문서의 워크시트를 엽니다
2. 메뉴바에서 **"연동"** 버튼을 클릭합니다
3. 연동 대상 문서를 선택합니다
4. 확인 팝업에서 **"연동 실행"**을 클릭합니다
5. 연동 완료 메시지를 확인합니다

## 3. 개정 출처 마커

### 3.1 마커란?

연동을 통해 데이터가 전달되면, 전달받은 항목 끝에 작은 위첨자 기호가 붙습니다. 이 기호를 **개정 출처 마커**라고 합니다. 편지 봉투의 발신자 스탬프처럼, 이 데이터가 어디서 왔는지 한눈에 알려줍니다.

### 3.2 마커 종류

| 마커 | 색상 | 출처 | 의미 | 예시 |
|------|------|------|------|------|
| **ᶠ** | 파란색 (`#1565c0`) | PFMEA 또는 DFMEA | FMEA에서 연동/개정된 항목 | `용접 전류ᶠ` |
| **ᶜ** | 녹색 (`#2e7d32`) | CP (관리계획서) | CP에서 연동/개정된 항목 | `저항 용접ᶜ` |
| **ᵖ** | 주황색 (`#e65100`) | PFD (공정흐름도) | PFD에서 연동/개정된 항목 | `프레스 가공ᵖ` |

### 3.3 마커 동작 규칙

| 규칙 | 설명 |
|------|------|
| 자동 부착 | 연동 실행 시 전달된 데이터에 출처 마커가 자동으로 붙습니다 |
| 중복 방지 | 동일 출처 마커가 이미 있으면 다시 붙지 않습니다 |
| 추적 가능 | 마커를 보면 어디서 데이터가 변경/전달되었는지 즉시 파악할 수 있습니다 |

### 3.4 마커 활용 예시

**시나리오**: CP에서 "저항 용접" 공정을 수정한 후 PFD와 PFMEA로 연동한 경우

```
[CP 워크시트]     저항 용접           ← 원본 (마커 없음)
[PFD 워크시트]    저항 용접ᶜ          ← CP에서 전달됨 (녹색 ᶜ)
[PFMEA 워크시트]  저항 용접ᶜ          ← CP에서 전달됨 (녹색 ᶜ)
```

**시나리오**: PFMEA에서 공정특성을 수정한 후 PFD로 연동한 경우

```
[PFMEA 워크시트]  용접 전류 15A       ← 원본 (마커 없음)
[PFD 워크시트]    용접 전류 15Aᶠ      ← FMEA에서 전달됨 (파란색 ᶠ)
```

### 3.5 마커 제거

마커가 포함된 텍스트를 직접 수정하면 마커가 유지됩니다. 마커를 제거하려면 텍스트에서 위첨자 기호(ᶠ, ᶜ, ᵖ)를 수동으로 삭제하면 됩니다.

## 4. 연동 데이터 무결성

### 4.1 무결성이란?

연동 과정에서 데이터가 변질되거나 누락되지 않는 것을 **데이터 무결성**이라고 합니다. 복사기에 문서를 넣으면 원본과 동일한 사본이 나오는 것과 같은 원리입니다.

### 4.2 보장 항목

| 항목 | 보장 내용 | 설명 |
|------|----------|------|
| **왕복 연동 (Round-Trip)** | 데이터 열화 없음 | A→B→A로 왕복 연동해도 원본 데이터와 동일합니다 |
| **특별특성 보존** | CC/SC 값 유지 | 특별특성 코드(CC=중요특성, SC=안전특성)가 연동 과정에서 변환되지만 의미는 동일하게 보존됩니다 |
| **한국어/특수문자 보존** | 문자 깨짐 없음 | 한글, 특수문자, 유니코드 문자가 연동 후에도 정확히 유지됩니다 |
| **정렬순서 유지** | sortOrder 보존 | 공정의 표시 순서가 연동 후에도 원본과 동일하게 유지됩니다 |
| **셀병합 유지** | 병합 상태 보존 | 같은 값을 가진 연속 행의 셀병합 상태가 연동 후에도 유지됩니다 |

### 4.3 특별특성 변환 규칙

FMEA와 PFD/CP 간에 특별특성 코드 표기가 약간 다릅니다. 연동 시 자동으로 변환됩니다.

| FMEA 표기 | 연동 후 PFD/CP 표기 | 의미 |
|-----------|-------------------|------|
| `C` 또는 `CC` | `CC` | 중요특성 (Critical Characteristic) |
| `S` 또는 `SC` | `SC` | 안전특성 (Safety Characteristic) |
| `I` 또는 `IC` | `IC` | 중요 항목 (Important Characteristic) |

### 4.4 주의사항

| 상황 | 주의점 |
|------|--------|
| 동시 수정 | 두 사용자가 같은 문서를 동시에 수정하고 각각 다른 문서로 연동하면 마지막 연동이 적용됩니다 |
| 1L 데이터 | FMEA의 1L(완제품) 영역에서 구분(YP/SP/EU), 기능, 요구사항은 연동되지 않습니다. FMEA에서 직접 입력해야 합니다 |
| 개정 전 연동 | 개정을 시작하기 전에 먼저 연동을 실행하면, 최신 데이터로 개정 작업을 시작할 수 있습니다 |

## 5. 실무 시나리오

### 시나리오 1: 신규 프로젝트 — CP 기준으로 FMEA/PFD 생성

1. **CP 등록 및 Import** — 관리계획서를 먼저 작성합니다
2. **CP → PFD 연동** — CP 데이터를 PFD로 전달하여 공정흐름도를 자동 생성합니다
3. **CP → PFMEA 연동** — CP 데이터를 PFMEA로 전달하여 구조분석을 자동 생성합니다
4. **PFMEA 워크시트 작업** — 고장분석, 위험평가, 개선활동을 PFMEA에서 진행합니다

### 시나리오 2: PFMEA 수정 후 PFD/CP 업데이트

1. **PFMEA 워크시트 수정** — 공정 정보를 변경합니다
2. **PFMEA → PFD 연동** — 변경된 공정 정보를 PFD에 반영합니다
3. **PFD → CP 연동** — PFD를 통해 CP에도 반영합니다
4. **마커 확인** — PFD와 CP에서 ᶠ(파란색) 마커로 FMEA 출처 항목을 확인합니다

### 시나리오 3: 개정 관리 흐름

1. 개정관리 화면에서 **"REV N+1"** 클릭 → 새 개정 시작
2. 문서 ID의 개정 접미사가 자동 증가 (예: `-r00` → `-r01`)
3. 워크시트에서 필요한 내용을 수정합니다
4. 연동 문서가 있으면 연동을 실행하여 다른 문서에도 반영합니다
5. 개정관리 화면에서 결재(상신→검토→승인)를 진행합니다
