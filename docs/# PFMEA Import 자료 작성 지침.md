## 1. 파일 구조 개요

Import 엑셀 파일은 5개 시트로 구성된다.

| 시트명 | 행수 | 역할 |
|--------|------|------|
| `L1 통합(C1-C4)` | 20행 | 시스템 레벨 — 제품기능·요구사항·고장영향 |
| `L2 통합(A1-A6)` | 28행 | 공정 레벨 — 공정·제품특성·고장형태·검출관리 |
| `L3 통합(B1-B5)` | 115행 | 요소 레벨 — 작업요소·기능·공정특성·고장원인·예방관리 |
| `FC 고장사슬` | 115행 | 고장사슬 — FE·FM·FC N:1:N 연결 |
| `FA 통합분석` | 참조 | 통합 분석 뷰 (SOD·AP·LLD 추천) |

## 2. UUID 및 FK 설계 원칙

### 2-1. UUID 생성 기준

| 구분 | UUID 생성 기준 | 설명 |
|------|---------------|------|
| 고유 텍스트 기반 | 텍스트 고유값 수만큼 UUID 생성 | 중복 행에서는 기존 UUID 재사용 |
| 복합키 기반 (L3) | 공정번호+상위필드+해당필드 조합 | 같은 텍스트라도 공정이 다르면 별도 UUID |
| 행 기반 (FC만) | 115행 전체 각각 UUID | 링크 레코드 — 텍스트 중복과 무관하게 매행 신규 |

### 2-2. parentId 연결 구조

```
C1 (루트)
└─ C2.parentId = C1.uuid
   └─ C3.parentId = C2.uuid
      └─ C4.parentId = C3.uuid          ← 고장영향

A1 (루트)
├─ A4.parentId = A1.uuid
│  └─ A5.parentId = A4.uuid             ← 고장형태
│     └─ A6.parentId = A5.uuid          ← 검출관리
└─ B1.parentId = A1.uuid (공정번호 연결)
   └─ B2.parentId = B1.uuid
      └─ B3.parentId = B2.uuid
         └─ B4.parentId = B3.uuid       ← 고장원인
            └─ B5.parentId = B4.uuid    ← 예방관리
```

### 2-3. FK 텍스트 완전일치 규칙

파서는 텍스트 완전일치로 UUID를 조회한다. 아래 3쌍은 반드시 일치해야 한다.

| FC 필드 | 참조 시트·필드 | 비고 |
|---------|--------------|------|
| `FE(고장영향)` | L1 `C4 고장영향` | forward-fill 후 비교 |
| `FM(고장형태)` | L2 `A5 고장형태` | forward-fill 후 비교 |
| `FC(고장원인)` | L3 `B4 고장원인` | 유효행 판별 기준 |
| `B5.예방관리` | L3 `B5 예방관리` | 텍스트 일치로 조회 |
| `A6.검출관리` | L2 `A6 검출관리` | 텍스트 일치로 조회 |

## 3. L1 시트 (C1~C4) 작성 규칙

**유효행 판별:** `구분(C1)` notna → **20행**

| 컬럼 | UUID수 | 행수 | UUID 키 | parentId | FK 제공 |
|------|--------|------|---------|----------|---------|
| C1 구분 | 3 | 20 | C1 텍스트 고유 | 없음(루트) | C2.parentId |
| C2 제품기능 | 7 | 20 | C2 텍스트 고유 | C1.uuid | C3.parentId |
| C3 요구사항 | 17 | 20 | C3 텍스트 고유 | C2.uuid | C4.parentId |
| **C4 고장영향** ★ | **20** | **20** | C4 텍스트 고유 | C3.uuid | FC FE → C4.uuid |

### 작성 규칙

- **C4 고장영향은 행수 = 고유수 = UUID수 (20개)** — 중복 시 파서 오류
- C1~C3은 중복 허용 (상위 항목이 복수 하위에 반복되는 정상 구조)
- C4는 FC 고장사슬 FE의 FK 대상이므로 텍스트 오타·공백 불일치 절대 금지

## 4. L2 시트 (A1~A6) 작성 규칙

**유효행 판별:** `A1.공정번호` notna → **28행**

| 컬럼 | UUID수 | 행수 | UUID 키 | parentId | FK 제공 |
|------|--------|------|---------|----------|---------|
| A1 공정번호 | 17 | 28 | 공정번호 고유 | 없음(루트) | A4.parentId, B1.parentId |
| A2 공정명 | 17 | 28 | 공정번호 고유 | A1.uuid | — (A1 속성) |
| A3 공정기능 | 20 | 28 | A3 텍스트 고유 | A1.uuid | — (A1 속성) |
| A4 제품특성 | 28 | 28 | A4 텍스트 고유 | A1.uuid | A5.parentId |
| **A5 고장형태** ★ | **28** | **28** | A5 텍스트 고유 | A4.uuid | FC FM → A5.uuid |
| A6 검출관리 | 17 | 28 | A6 텍스트 고유 | A5.uuid | FC A6 → A6.uuid |

### 작성 규칙

- **A5 고장형태는 행수 = 고유수 = UUID수 (28개)** — 중복 시 파서 오류
- A4 제품특성도 행수 = 고유수 (28개) — A5와 1:1 대응
- A1 공정번호 중복(17고유/28행)은 복수 FM 보유 공정에서 발생하는 정상 구조
  - 복수 FM 보유 공정: 10·20·40·100·140·150
- A6 검출관리 중복(17고유/28행)은 복수 FM이 동일 DC를 공유하는 정상 구조
- 특별특성 컬럼: CC(Critical Characteristic) 또는 SC(Significant Characteristic) 해당 시 기재, 없으면 공란

## 5. L3 시트 (B1~B5) 작성 규칙

**유효행 판별:** `공정번호` notna → **115행**  
**행 1개의 의미:** 공정번호 + 작업요소(B1) + 고장원인(B4) 조합 1개 = 독립 레코드

| 컬럼 | UUID수 | 행수 | UUID 키 | parentId | FK 제공 |
|------|--------|------|---------|----------|---------|
| B1 작업요소 | 91 | 115 | 공정번호+B1 복합키 | A1.uuid | B2.parentId |
| **B2 요소기능** ★ | **91** | **115** | 공정번호+B1+B2 복합키 | B1.uuid | B3.parentId |
| B3 공정특성 | 103 | 115 | 공정번호+B1+B2+B3 복합키 | B2.uuid | B4.parentId |
| **B4 고장원인** ★ | **113** | **115** | 공정번호+B1+B4 복합키 | B3.uuid | FC FC → B4.uuid, B5.parentId |
| B5 예방관리 | 113 | 115 | 공정번호+B1+B4+B5 복합키 | B4.uuid | FC B5 → B5.uuid |

### 작성 규칙

#### 5-1. 복합키 원칙
- **L3의 모든 UUID는 공정번호를 포함한 복합키로 생성**
- 텍스트가 같아도 공정번호가 다르면 별도 UUID
- 예: `Sorter 장비`(공정20)와 `Sorter 장비`(공정180)는 다른 B1 UUID

#### 5-2. 계층 관계 (1:N)
- B1 : B2 = 1 : 1 (작업요소 1개당 요소기능 1개)
- B2 : B3 = 1 : N (요소기능 1개가 복수 공정특성 보유 가능, 최대 5개)
- B3 : B4 = 1 : N (공정특성 1개에 복수 고장원인 연결 가능, 최대 4개)
  - 예: 온도(B3) → 온도 초과 / 온도 미달 / 온도 불안정 (B4 3개)
- B4 : B5 = 1 : N (고장원인 1개에 복수 예방관리 가능, 최대 2개)

#### 5-3. 중복 발생 이유 (정상 구조)
| 필드 | 텍스트 고유 | 복합키 UUID | 중복 이유 |
|------|-----------|------------|---------|
| B1 작업요소 | 81 | 91 | 같은 요소명이 다른 공정에 존재 |
| B2 요소기능 | 88 | 91 | B1과 동일 패턴 |
| B3 공정특성 | 50 | 103 | B3:B4=1:N으로 B4 수만큼 행 반복 |
| B4 고장원인 | 42 | 113 | 같은 원인이 다른 공정·특성에 존재 |
| B5 예방관리 | 81 | 113 | B4:B5=1:N 가능 |

#### 5-4. 4M 분류 코드
| 코드 | 의미 |
|------|------|
| MN | Man — 작업자 |
| MC | Machine — 설비 |
| MT | Material — 자재 |
| EN | Environment — 환경 |

#### 5-5. 특별특성 컬럼
B3 공정특성에 대한 CC/SC 구분. 해당 없으면 공란.

## 6. FC 고장사슬 시트 작성 규칙

**유효행 판별:** `FC(고장원인)` notna → **115행**  
**구조:** FE(N) ← FM(1) → FC(N) = N:1:N

### 6-1. 컬럼 정의

| 컬럼 | UUID | parentId | FK | 기재 규칙 |
|------|------|----------|----|---------|
| FE구분 | 없음 | — | — | 첫행만 기재 |
| **FE 고장영향** | 없음 | — | → C4.uuid | 첫행만 기재, forward-fill 처리 |
| L2-1.공정번호 | 없음 | — | — | 첫행만 기재, forward-fill 처리 |
| **FM 고장형태** ★ | 없음 | — | → A5.uuid | 첫행만 기재, forward-fill 처리 |
| 4M | 없음 | — | — | 매행 기재 (L3 4M과 동일) |
| 작업요소(WE) | 없음 | — | — | 매행 기재 (L3 B1과 일치) |
| **FC 고장원인** ★ | **115개** | → A5.uuid | → B4.uuid | **매행 기재 필수** |
| B5.예방관리 | 없음 | — | → B5.uuid | 매행 기재 |
| A6.검출관리 | 없음 | — | → A6.uuid | 매행 기재 |
| O 발생도 | 없음 | — | — | FC 레코드 속성 (1~10) |
| D 검출도 | 없음 | — | — | FC 레코드 속성 (1~10) |
| AP | 없음 | — | — | FC 레코드 속성 (H/M/L) |

### 6-2. forward-fill 규칙

FE·FM·공정번호는 그룹의 **첫 번째 행에만 기재**하고 이하 행은 공란으로 둔다.  
파서는 반드시 forward-fill을 적용한 후 FK 조회를 수행해야 한다.

```
행1: 공정01 | 파티클 초과 | 작업 숙련도 부족   ← FM 기재
행2:        |            | 온습도 제어 이탈   ← FM 공란 → forward-fill로 채움
행3:        |            | 풍속 저하          ← FM 공란 → forward-fill로 채움
행4: 공정10 | TTV 규격 초과 | 측정 오차        ← 새 FM 기재
```

### 6-3. FC UUID 생성 특칙

FC 고장원인(FC 컬럼)은 **링크 레코드**이므로 텍스트 중복과 무관하게 115행 전체에 각각 신규 UUID를 생성한다.  
같은 `FC(고장원인)` 텍스트가 여러 행에 있어도 각 행이 독립 UUID를 갖는다.

### 6-4. 115행 구성 필수 요건

L3 B4 고장원인 115행 전체에 FC가 연결되어야 누락이 없다.

- FC 시트의 `FC(고장원인)` 값 = L3 `B4 고장원인` 값 (텍스트 완전일치)
- L3에만 있고 FC에 없는 B4가 존재하면 파서에서 FK 연결 실패
- 검증: `set(L3.B4) - set(FC.FC(고장원인))` = 공집합이어야 함

## 7. 파서 구현 지침

### 7-1. 시트별 처리 순서

```
1. L1 파싱   → C1→C2→C3→C4 UUID 생성
2. L2 파싱   → A1→A4→A5→A6 UUID 생성
3. L3 파싱   → B1→B2→B3→B4→B5 UUID 생성 (복합키 기준)
4. FC 파싱   → forward-fill → FK 조회 → 링크 레코드 UUID 생성
5. FA 파싱   → 참조 뷰 (UUID 생성 없음)
```

### 7-2. 유효행 판별 기준

| 시트 | 판별 컬럼 | 유효행 수 |
|------|----------|---------|
| L1 | `구분(C1)` notna | 20행 |
| L2 | `A1.공정번호` notna | 28행 |
| L3 | `공정번호` notna | 115행 |
| FC | `FC(고장원인)` notna | 115행 |

### 7-3. 중복 처리 규칙

```python
# UUID 재사용 패턴 (L1·L2·L3 공통)
uuid_cache = {}
for row in rows:
    key = make_key(row)          # 텍스트 또는 복합키
    if key not in uuid_cache:
        uuid_cache[key] = new_uuid()
    record.uuid = uuid_cache[key]

# FC 고장원인만 예외 — 매행 신규
for row in fc_rows:
    fc_record.uuid = new_uuid()  # 캐시 없이 무조건 신규
```

### 7-4. FK 조회 실패 처리

```
FK 조회 실패 조건:
  - FC FE 텍스트가 L1 C4에 없음
  - FC FM 텍스트가 L2 A5에 없음
  - FC FC 텍스트가 L3 B4에 없음

처리 방법:
  - 오류 로그 기록: {시트}.{컬럼} = "{텍스트}" → FK 미매칭
  - 해당 행 import 중단 (고아 레코드 방지)
  - 오류 행 목록 반환 후 사용자 확인 요청
```

### 7-5. 데이터 무결성 검증 체크리스트

파서 실행 전 사전 검증:

```
[ ] L1 C4 고장영향: 행수(20) = 고유수(20) — 중복 시 오류
[ ] L2 A5 고장형태: 행수(28) = 고유수(28) — 중복 시 오류
[ ] L2 A4 제품특성: 행수(28) = 고유수(28) — 중복 시 오류
[ ] FC FM forward-fill 후 L2 A5와 완전 매칭
[ ] FC FE forward-fill 후 L1 C4와 완전 매칭
[ ] FC FC 전체가 L3 B4와 완전 매칭 (누락 = 0)
[ ] set(L3.B4 텍스트) - set(FC.FC 텍스트) = 공집합
[ ] L3 행수 = 115
[ ] FC 유효행수 = 115
```

## 8. 오류 유형 및 원인

| 오류 | 원인 | 해결 |
|------|------|------|
| FK 연결 실패 (FM→A5) | FC FM 텍스트 ≠ L2 A5 텍스트 | 텍스트 완전일치 수정 |
| FK 연결 실패 (FC→B4) | FC FC 텍스트 ≠ L3 B4 텍스트 | L3 B4 추가 또는 FC 수정 |
| C4 중복 UUID 충돌 | L1 C4에 동일 고장영향 2행 | 중복행 삭제 |
| A5 중복 UUID 충돌 | L2 A5에 동일 고장형태 2행 | 중복행 삭제 |
| FC 고아 레코드 | L3에 없는 B4가 FC에 존재 | L3에 B4 추가 또는 FC 삭제 |
| B4 누락 | L3에 있는 B4가 FC에 없음 | FC에 해당 행 추가 |
| forward-fill 오류 | FM/FE 기재행이 잘못된 위치 | FM 그룹 첫행에만 기재 확인 |

## 9. 컬럼별 텍스트 작성 규칙

### 공통

- 셀 내 줄바꿈(`\n`) 사용 금지 — 파서 분리 오류 발생
- 앞뒤 공백 제거 — FK 조회 시 공백 포함 텍스트는 불일치 처리됨
- 특수문자 통일: `·`(가운뎃점), `~`(물결), `±`, `μm`, `℃`, `%` 등 동일 문자 사용
- 복수 항목 구분자: 쉼표+공백 (`, `) 통일

### B5 예방관리 / A6 검출관리 복수 항목

```
올바른 예: 정류기(Rectifier) 정기 교정, 출력 전류 모니터링
잘못된 예: 정류기(Rectifier) 정기 교정 / 출력 전류 모니터링  (구분자 불일치)
잘못된 예: 정류기(Rectifier) 정기 교정\n출력 전류 모니터링   (줄바꿈 사용)
```
