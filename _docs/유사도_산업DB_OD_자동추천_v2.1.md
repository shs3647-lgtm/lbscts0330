## 1. 시스템 개요

PC1(예방관리) / DC1(검출관리) 자동 채움 시, 3가지 데이터 소스를 복합 활용하여 **가장 효과적인(O/D가 가장 낮은)** 관리방법을 추천합니다.

| 순서 | 데이터 소스 | 설명 |
|------|-----------|------|
| 1 | **Import 데이터** (B5/A6) | 기초정보 엑셀에서 import한 원본 데이터. processNo 기준 매칭 |
| 2 | **유사도 매칭** | FC/FM 텍스트와 B5/A6 후보 간 한국제조업 도메인 유사도 점수 기반 랭킹 |
| 3 | **한국산업 공통DB** | `KrIndustryPrevention` / `KrIndustryDetection` 테이블의 산업 표준 관리방법 |

**핵심 원칙**:
- **발생도(O)**: 예방관리 기준 — 더 낮은 O를 주는 PC를 선택
- **검출도(D)**: 성숙도 기준 — 더 낮은 D를 주는 DC를 선택
- Import보다 효과가 큰(O/D가 낮은) 후보가 있으면 **그것을 추천**

### ★ v2.1 절대 규칙: 추천값 상한선

> **⚠️ CRITICAL**: 자동추천 시스템이 산출하는 O/D는 반드시 **4 이하**여야 합니다.
>
> | 구분 | 상한선 | 근거 |
> |------|--------|------|
> | **발생도(O)** | **O ≤ 4** | 추천하는 예방관리는 최소 "체계적 관리" 수준 이상이어야 함 |
> | **검출도(D)** | **D ≤ 4** | 추천하는 검출관리는 최소 "자동판정/계측장비" 수준 이상이어야 함 |
>
> **O≥5 또는 D≥5인 후보는 추천 대상에서 제외**합니다.
> 추천 가능한 후보가 없으면 "추천 없음"으로 표시하고, 사용자가 직접 선택하도록 합니다.

## 3. 발생도(O) 평가 체계 (★ v2.1 전면 개편)

### 3.1 평가 기준 (AIAG-VDA P-FMEA — v2.1 보정)

> **변경 철학**: AIAG-VDA 원본 기준에서 한국 제조업 실무를 반영하여 보정.
> "교육/표준"도 **체계적으로 수립·운영**되면 O=4 이하로 인정.
> 핵심은 **"관리의 체계성"** — 단순 교육이 아니라 시스템적 관리 여부로 판별.

| O값 | 관리 수준 | 예방관리(PC) 키워드 예시 | v2.1 변경사항 |
|-----|----------|-------------------------|-------------|
| **1** | 고장 원천 제거 | 설계변경, 구조변경, 원인 제거 | (변경 없음) |
| **2** | 에러프루프 (물리적 차단) | Poka-Yoke, 인터록, 자동차단, 라인정지 | (변경 없음) |
| **3** | 자동식별 + 체계적 계측관리 | 바코드, QR스캔, RFID, 비전검사, **검교정 주기관리**, **MSA** | ★ 검교정·MSA를 PC O=3으로 이동 |
| **4** | 체계적 관리 (표준+시스템) | SPC, 토크관리, PM, 파라미터 관리, **작업표준서(WI)**, **기술표준**, **이종자재 식별교육+식별시스템**, **역량인증+절차서** | ★ 작업표준/기술표준/식별교육을 O=4로 상향 |
| ~~5~~ | ~~표준화된 관리~~ | ~~역량인증, 작업표준, 절차서~~ | ❌ **삭제** — O=4로 통합 |
| ~~6~~ | ~~기본 인적 관리~~ | ~~교육, 훈련, OJT~~ | ❌ **삭제** — 목적별로 O=3~4에 분산 |
| **5~10** | 관리 미비/미수립 | (키워드 미매칭 또는 비체계적 관리) | **추천 대상 제외** |

### 3.2 교육·표준 세분화 규칙 (★ v2.1 신규)

> **핵심 원칙**: "교육"이라는 단어만으로 일률 O=6을 부여하던 방식 폐지.
> 교육/표준의 **목적과 체계성**에 따라 O=3~4로 차등 배정.

| 교육/표준 유형 | O값 | 판별 키워드 | 근거 |
|--------------|-----|-----------|------|
| **자동식별 연계 교육** | **3** | "바코드 스캔 교육", "QR 스캔 확인 교육", "비전검사 운용 교육" | 자동식별 시스템이 전제되므로 O=3 |
| **검교정·MSA 교육** | **3** | "검교정 교육", "MSA 교육", "Gage R&R 교육", "계측기 관리 교육" | 계측 체계 수립이 전제 |
| **이종자재 식별 교육** | **4** | "이종자재 식별", "이종품 식별교육", "자재 식별표시 관리", "혼입방지 교육" | 식별 시스템 + 교육 = 체계적 관리 |
| **작업표준서(WI) 교육** | **4** | "작업표준서 교육", "WI 교육", "작업표준 교육훈련" | 표준 문서화가 전제 |
| **기술표준 관리** | **4** | "기술표준", "공정표준", "품질표준", "검사표준", "관리표준" | 표준 체계 수립 |
| **역량인증 제도** | **4** | "역량인증", "자격인증", "숙련도 인증", "작업자 인증" | 체계적 인증 시스템 |
| **정기 교육/OJT** | **4** | "정기 교육", "OJT", "숙련도 평가", "교육훈련" | 체계적 교육 프로그램 |
| **체크리스트 기반 확인** | **4** | "체크리스트", "점검표", "확인표" | 체계적 점검 시스템 |

> **★ 판별 순서**: 키워드가 복수 매칭될 경우 **가장 낮은 O값**을 채택.
> 예: "바코드 스캔 확인 교육 + 작업표준서 교육" → O=3 (바코드 스캔이 O=3)

### 3.3 검교정(Calibration)의 예방관리 이동 (★ v2.1 핵심 변경)

> **이전**: 검교정은 DC(검출관리)의 관리활동으로 분류, D=6 부여
> **변경**: 검교정은 **PC(예방관리)**로 이동, O=3~4 부여

**근거**:
- 검교정(Calibration)은 "측정 장비가 정확하게 작동하도록 사전에 보전하는 활동"
- 이는 고장을 **사전에 예방**하는 활동이지, 고장을 **사후에 검출**하는 활동이 아님
- AIAG-VDA 기준에서도 검교정은 "예방적 관리"에 해당

| 검교정 유형 | 분류 | O/D | 키워드 |
|-----------|------|-----|--------|
| 계측기 정기 검교정 | **PC (예방관리)** | **O=3** | 검교정, Calibration, 교정주기, 정기교정 |
| MSA/Gage R&R 실시 | **PC (예방관리)** | **O=3** | MSA, Gage R&R, GR&R, 측정시스템분석 |
| 정도관리 (precision) | **PC (예방관리)** | **O=4** | 정도관리, 측정불확도, 계측정밀도 |
| 검교정 기록 관리 | **PC (예방관리)** | **O=4** | 교정이력, 검교정대장, 교정성적서 |

**코드 변경 영향**:
```
[이전] detectionRatingMap.ts → MANAGEMENT_KEYWORDS에서 "검교정, MSA" 필터링 → D=6
[변경] pcOccurrenceMap.ts → MATURITY_RULES에 "검교정, MSA" 추가 → O=3
       detectionRatingMap.ts → MANAGEMENT_KEYWORDS에서 "검교정, MSA" 제거
```

### 3.4 평가 함수 체계 (v2.1 업데이트)

```
evaluateO(pcText)
├── correctOccurrence(pcText)     ← pcOccurrenceMap.ts (1차: 직접 키워드 매칭)
│   └── MATURITY_RULES 16개 규칙 (★ 12개→16개 확장)
│       ├── O=2: Poka-Yoke, 인터록, 자동차단, 투입중지 ...
│       ├── O=3: 바코드, QR스캔, RFID, 자동검사, ★검교정, ★MSA, ★Gage R&R ...
│       ├── O=4: SPC, 토크관리, PM, ★작업표준서(WI), ★기술표준, ★역량인증,
│       │        ★이종자재 식별교육, ★정기교육, ★OJT, ★체크리스트, ★절차서 ...
│       └── (O≥5 규칙 삭제 — 추천 대상 제외)
│
├── ★ clampO(value)               ← v2.1 신규: 상한선 적용
│   └── return Math.min(value, 4)  // O > 4이면 추천 제외
│
└── recommendOccurrence(pcText)   ← occurrenceRecommendMap.ts (2차: 관리유형 분류)
    └── analyzePCControlType → O_RECOMMENDATION_TABLE (v2.1 보정)
        ├── designPrevent → O=1
        ├── bestPractice  → O=2~3 (★ 검교정·MSA 포함)
        ├── technical     → O=3~4 (★ 설비점검, 파라미터 관리)
        ├── systematic    → O=4   (★ v2.1 신규: 작업표준/기술표준/교육/체크리스트)
        └── none          → null  (★ 추천 제외 — O=5~10 범위 삭제)
```

**평가 우선순위**: `correctOccurrence` 결과가 유효하면 사용, 아니면 `recommendOccurrence` fallback.
결과가 null이면 추천 불가 → 사용자 직접 입력.

```
2. 전체 풀(Import + 산업DB)에서 각 후보의 O값 계산
3. ★ O > 4인 후보를 제거 (상한선 필터)
4. 남은 후보 중 findBestPC() → O가 가장 낮은 후보 선택
5. 각 failureLink에 대해:
   a. 기존 PC가 있으면 → 스킵 (기존값 유지)
   b. 해당 processNo의 Import B5 후보 중 O≤4만 대상
   c. Import 중 O가 가장 낮은 후보 선택
   d. 전체 풀에서 더 낮은 O 후보가 있으면 → 교체
   e. ★ 모든 후보가 O>4이면 → PC1 = "(추천 없음)", O = null
   f. 최종 PC 텍스트 + O값 → riskData에 저장
```
