# 수동 트리거 연동 방식 설계서

> **작성일**: 2026-01-29  
> **목적**: DB 중심의 수동 트리거 기반 연동 시스템 구축  
> **상태**: 📋 설계 제안

---

## 1. 현재 문제점

### 1.1 기존 자동 추론 방식의 문제

```
[현재 방식]
사용자가 PFMEA ID 입력
    ↓
useEffect에서 ID 패턴 분석
    ↓
PFD, CP ID 자동 생성 (추론)
    ↓
기초정보 자동 채움 시도
    ↓
❌ 문제: 실제 DB 데이터와 불일치 가능
```

| 문제 유형 | 설명 | 영향 |
|----------|------|------|
| **추론 오류** | ID 패턴 기반 추론이 실제 관계와 불일치 | 잘못된 데이터 연결 |
| **신규/편집 혼동** | 신규 생성 시에도 기존 데이터 로드 시도 | 빈 폼에 비정상 데이터 표시 |
| **연동 불일치** | 각 앱에서 개별 연동 → 불일치 발생 | 데이터 무결성 훼손 |
| **의도치 않은 덮어쓰기** | 자동 채움이 수동 입력 덮어씀 | 사용자 입력 손실 |

### 1.2 삭제된 자동 추론 로직

> 참조: `docs/자동추론검토.md`

- PFMEA: 연동 PFD/CP 자동 생성, 기초정보 자동 연동
- DFMEA: 기초정보 자동 연동
- CP: 상위 APQP/FMEA 자동 추론, 기초정보 자동 연동
- PFD: 연동 CP 자동 생성, 기초정보 자동 연동

### 2.3 연동 모드 vs 단독 모드

| 항목 | 연동 모드 (Linked) | 단독 모드 (Solo) |
|------|-------------------|-----------------|
| **ID 접미사** | `-L01`, `-L02`, ... | `-S` |
| **기초정보** | 모든 연동 앱이 공유 | 해당 앱에서만 입력 |
| **ProjectLinkage** | 레코드 생성 | 레코드 없음 |
| **앱 간 탐색** | 버튼으로 연동 앱 이동 가능 | 해당 없음 |
| **데이터 변경** | 마스터에서 변경 시 동기화 옵션 | 개별 관리 |

## 3. DB 스키마 개선

### 3.1 ProjectLinkage 테이블 (확장)

```prisma
model ProjectLinkage {
  id            String   @id @default(uuid())
  
  // 연동 그룹 관리
  linkGroupNo   Int      @default(1)      // ★ 연동 그룹 번호 (1~99)
  linkType      String   @default("auto") // 'auto' | 'manual'
  status        String   @default("active")
  
  // 프로젝트 공통 정보 (마스터)
  projectName   String?
  customerName  String?
  modelYear     String?
  companyName   String?
  
  // 연동 문서 ID들
  apqpNo        String?  // ★ APQP (최상위)
  pfmeaId       String?  // ★ PFMEA
  dfmeaId       String?  // ★ DFMEA
  pfdNo         String?  // ★ PFD
  cpNo          String?  // ★ CP
  
  // 생성 플래그 (어떤 앱을 생성할지)
  createApqp    Boolean  @default(false)
  createPfmea   Boolean  @default(false)
  createDfmea   Boolean  @default(false)
  createPfd     Boolean  @default(false)
  createCp      Boolean  @default(false)
  
  // 타임스탬프
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
  createdBy     String?
  
  @@map("project_linkages")
}
```

### 3.2 각 앱 Registration 테이블 (기존 유지 + 명시적 연동 필드)

```prisma
model PfmeaRegistration {
  fmeaId         String   @id
  
  // ★ 명시적 연동 필드 (추론 대신 DB 저장)

  linkedCpNos    String[] // 연동된 CP 목록
  
  // 기초정보 (프로젝트에서 복사 또는 개별 입력)
  companyName    String?
  fmeaName       String?
  projectName    String?
  customerName   String?
  // ... 나머지 필드
}
```

## 5. API 설계

### 5.1 프로젝트 생성 API (신규)

```typescript
// POST /api/project
// 프로젝트 및 연동 문서 일괄 생성

interface CreateProjectRequest {
  // 기본 정보
  projectName: string;
  customerName: string;
  modelYear: string;
  companyName?: string;
  
  // 연동 모드
  linkMode: 'linked' | 'solo';
  
  // 생성할 앱 목록
  apps: {
    apqp?: boolean;
    pfmea?: boolean;
    dfmea?: boolean;
    pfd?: boolean;
    cp?: boolean;
  };
}

interface CreateProjectResponse {
  success: boolean;
  linkGroupNo: number;
  createdDocs: {
    apqpNo?: string;
    pfmeaId?: string;
    dfmeaId?: string;
    pfdNo?: string;
    cpNo?: string;
  };
  linkageId: string;
}
```

### 5.2 기초정보 연동 API

```typescript
// POST /api/sync-basic-info
// 상위 문서에서 기초정보 가져오기 (수동 트리거)

  targetDocId: string;      // 대상 문서 ID (예: pfm26-m001-L01)
  sourceDocId: string;      // 원본 문서 ID (예: apqp26-m001-L01)
  syncFields?: string[];    // 동기화할 필드 목록 (없으면 전체)
}
```

## 9. 기대 효과

### 9.1 Before vs After

| 항목 | Before (자동 추론) | After (수동 트리거) |
|------|-------------------|-------------------|
| **데이터 정확성** | 추론 오류 가능 | DB 데이터만 사용, 정확 |
| **사용자 경험** | 예기치 않은 자동 채움 | 명시적 제어, 예측 가능 |
| **연동 관리** | 각 앱 개별 처리 | 프로젝트 허브에서 통합 관리 |
| **유지보수** | 복잡한 추론 로직 | 단순한 DB 조회/저장 |
| **확장성** | 새 앱 추가 시 추론 로직 수정 | 프로젝트 허브에 앱 추가만 |

### 9.2 핵심 이점

1. **"추론하지 말고, DB에 저장하고 DB에서 로드하라"** 원칙 실현
2. 프로젝트 단위 통합 관리로 가시성 향상
3. 수동 트리거로 사용자 의도 명확화
4. 연동 그룹 번호로 복수 프로젝트 구분 용이
