# 유사도 · 산업DB 기반 발생도(O) · 검출도(D) 자동추천 시스템

> **작성일**: 2026-02-05  
> **최종 수정**: 2026-02-25

---

## 1. 시스템 개요

PC1(예방관리) / DC1(검출관리) 자동 채움 시, 3가지 데이터 소스를 복합 활용하여 **가장 효과적인(O/D가 가장 낮은)** 관리방법을 추천합니다.

| 순서 | 데이터 소스 | 설명 |
|------|-----------|------|
| 1 | **Import 데이터** (B5/A6) | 기초정보 엑셀에서 import한 원본 데이터. processNo 기준 매칭 |
| 2 | **유사도 매칭** | FC/FM 텍스트와 B5/A6 후보 간 한국제조업 도메인 유사도 점수 기반 랭킹 |
| 3 | **한국산업 공통DB** | `KrIndustryPrevention` / `KrIndustryDetection` 테이블의 산업 표준 관리방법 |

**핵심 원칙**:
- **발생도(O)**: 예방관리 기준 — 더 낮은 O를 주는 PC를 선택
- **검출도(D)**: 성숙도 기준 — 더 낮은 D를 주는 DC를 선택
- Import보다 효과가 큰(O/D가 낮은) 후보가 있으면 **그것을 추천**

## 3. 발생도(O) 평가 체계

### 3.1 평가 기준 (AIAG-VDA P-FMEA)

| O값 | 관리 수준 | 예방관리(PC) 키워드 예시 |
|-----|----------|-------------------------|
| **1** | 고장 원천 제거 | 설계변경, 구조변경, 원인 제거 |
| **2** | 에러프루프 (물리적 차단) | Poka-Yoke, 인터록, 자동차단, 라인정지 |
| **3** | 자동식별 (투입 방지) | 바코드, QR스캔, RFID, 비전검사, 자동검사 |
| **4** | 설비적/계측적 관리 | SPC, 검교정, 토크관리, 예방보전(PM), 파라미터 관리 |
| **5** | 표준화된 관리 | 역량인증, 작업표준, 절차서, 체크리스트 |
| **6** | 기본 인적 관리 | 교육, 숙련, 훈련, OJT |
| **7~10** | 관리 미비/미수립 | (텍스트 없음 또는 키워드 미매칭) |

### 3.2 평가 함수 체계

```
evaluateO(pcText)
├── correctOccurrence(pcText)     ← pcOccurrenceMap.ts (1차: 직접 키워드 매칭)
│   └── MATURITY_RULES 12개 규칙
│       ├── O=2: Poka-Yoke, 인터록, 자동차단, 투입중지 ...
│       ├── O=3: 바코드, QR스캔, RFID, 자동검사 ...
│       ├── O=4: SPC, 검교정, 토크관리, 예방보전 ...
│       ├── O=5: 역량인증, 작업표준, 절차서 ...
│       └── O=6: 교육, 훈련, OJT ...
│
└── recommendOccurrence(pcText)   ← occurrenceRecommendMap.ts (2차: 관리유형 분류)
    └── analyzePCControlType → O_RECOMMENDATION_TABLE
        ├── designPrevent → O=1
        ├── bestPractice  → O=2~3
        ├── technical     → O=4~5
        ├── behavioral    → O=6~7 (다소 효과적) / O=8~9 (거의 무효)
        └── none          → O=10
```

**평가 우선순위**: `correctOccurrence` 결과가 유효하면 사용, 아니면 `recommendOccurrence` fallback

### 3.3 PC 관리유형 분류 키워드

| 관리유형 | 키워드 | O 범위 |
|---------|--------|--------|
| `designPrevent` | 설계변경, 형상변경, 구조변경, 물리적 방지, 원인 제거 | 1 |
| `bestPractice` | SPC, 관리도, Cpk, 교정, PM, TPM, 지그설계, MSA, Gage R&R, FMEA, 공정능력 | 2~3 |
| `technical` | 센서, 자동, PLC, 모니터링, Poka-Yoke, 인터록, 바코드, 비전, 카메라 | 4~5 |
| `behavioral` | 교육, 훈련, 숙련도, OJT, 순회점검, 자주검사, 체크리스트, 확인, 점검 | 6~9 |

## 4. 검출도(D) 평가 체계

### 4.1 평가 기준 (AIAG-VDA P-FMEA)

| D값 | 검출능력 | 검출관리(DC) 키워드 예시 |
|-----|---------|-------------------------|
| **1** | 검출 불필요 | (에러프루프로 원인 자체 차단 — 자동 판정 불가) |
| **2** | 거의 확실히 검출 | Poka-Yoke, 인터록, 자동차단, 라인정지 |
| **3** | 높은 검출 (자동배출) | 바코드 스캐너, PDA 스캔, RFID |
| **4** | 높은 검출 (자동판정) | EOL, 비전검사, CMM, 체결검사기, 통전검사, 기능검사기, AOI, X-ray |
| **5** | 보통 (교정공구) | 토크렌치, 환경측정기, 온습도계 |
| **6** | 보통-낮음 (관리활동만) | 검교정, MSA, Gage R&R만 있고 실제 장비 없을 때 |
| **7** | 낮음 (수동검출) | 캘리퍼, 마이크로미터, 핀게이지, 자주검사 |
| **8** | 매우 낮음 (주관적) | 육안검사, 외관검사, 관능검사 |
| **9~10** | 극히 낮음/불가 | (키워드 미매칭) |

### 4.2 관리활동 필터링

다음 키워드는 **검출방법이 아닌 관리활동**으로 분류되어 D값 계산에서 제외됩니다:

| 관리활동 키워드 | 설명 |
|---------------|------|
| 검교정, 교정, Calibration | 측정기 교정 활동 |
| MSA, Gage R&R, GR&R | 측정시스템 분석 활동 |
| 정도관리 | 측정 정밀도 관리 활동 |

> **중요**: 관리활동만 있고 실제 장비가 없는 경우 D=6 부여

### 4.3 평가 함수

```
evaluateD(dcText) → recommendDetection(dcText)
├── 텍스트를 줄 단위 분리
├── 각 줄에서 "D:" 접두사 제거
├── 관리활동 키워드 → 필터링 (MANAGEMENT_KEYWORDS)
├── 실제 장비만 남김 → EQUIPMENT_RULES로 D값 매칭
└── 복수 장비 시 가장 낮은(최적) D값 채택
```

## 5. 유사도 점수 시스템

### 5.1 6가지 기법 가중 합산

| 기법 | 가중치 | 설명 |
|------|--------|------|
| 설비명/공구명 직접 매칭 | **25%** | FC의 "지그" → PC의 "지그 점검" 직접 매칭 |

## 8. 한국산업 공통DB (KrIndustry)

### 8.1 DB 스키마

```
KrIndustryPrevention
├── id, category, fcKeyword, method
├── m4Category (MN/MC/IM/EN)
├── description, sortOrder, isActive

KrIndustryDetection
├── id, category, fmKeyword, method
├── methodType (자동/수동/관리)
├── description, sortOrder, isActive
```

### 8.2 API

```
GET /api/kr-industry?type=prevention  → { prevention: [...] }
GET /api/kr-industry?type=detection   → { detection: [...] }
GET /api/kr-industry?type=all         → { prevention: [...], detection: [...] }
```

### 8.3 산업DB 활용 전략

| 구분 | Import 데이터만 사용 | Import + 산업DB 최적화 |
|------|--------------------|-----------------------|
| PC 선택 | B5의 첫번째 항목 사용 | B5 + 산업DB 풀에서 **O가 가장 낮은** 항목 선택 |
| DC 선택 | A6의 첫번째 항목 사용 | A6 + 산업DB 풀에서 **D가 가장 낮은** 항목 선택 |
| O 평가 | 선택된 PC 기반 | 선택 전에 O값을 미리 평가하여 최적 PC 선택 |
| D 평가 | 선택된 DC 기반 | 선택 전에 D값을 미리 평가하여 최적 DC 선택 |

## 9. fillPCDCFromImport 핵심 로직

### 9.1 PC1 선택 알고리즘 (발생도 기준)

```
1. Import B5 후보를 processNo로 그룹화
2. 전체 풀(Import + 산업DB)에서 O값 최적 후보 미리 계산 → findBestPC()
3. 각 failureLink에 대해:
   a. 기존 PC가 있으면 → 스킵 (기존값 유지)
   b. 해당 processNo의 Import B5 후보 각각에 evaluateO() 실행
   c. Import 중 O가 가장 낮은 후보 선택
   d. 전체 풀에서 더 낮은 O 후보가 있으면 → 교체
   e. 최종 PC 텍스트 + O값 → riskData에 저장
```

### 9.2 DC1 선택 알고리즘 (성숙도 기준)

```
1. Import A6 후보를 processNo로 그룹화
2. 전체 풀(Import + 산업DB)에서 D값 최적 후보 미리 계산 → findBestDC()
3. 각 failureLink에 대해:
   a. 기존 DC가 있으면 → 스킵 (기존값 유지)
   b. 해당 processNo의 Import A6 후보 각각에 evaluateD() 실행
   c. Import 중 D가 가장 낮은 후보 선택
   d. 전체 풀에서 더 낮은 D 후보가 있으면 → 교체
   e. 최종 DC 텍스트 + D값 → riskData에 저장
```

## 10. 자동추천 훅 (수동 실행)

`fillPCDCFromImport`는 자동채움(mount 시)이고, 수동 실행 훅은 별도입니다.

### 10.1 useAutoRecommendPC (예방관리)

| 항목 | 설명 |
|------|------|
| 소스 파일 | `hooks/useAutoRecommendPC.ts` |
| 매칭 전략 | processNo + m4 복합키 → 3단계 우선순위 |
| 랭킹 | `rankByPreventionRules()` — 유사도(40%) + 4M(15%) + FC규칙(25%) + FM규칙(20%) |
| O 평가 | `correctOccurrence()` 직접 산정 |
| 산업DB | `KrIndustryPrevention` → m4 기반 cross-process fallback |

### 10.2 useAutoRecommendDC (검출관리)

| 항목 | 설명 |
|------|------|
| 소스 파일 | `hooks/useAutoRecommendDC.ts` |
| 매칭 전략 | 공정번호 → FM키워드 규칙 → 유사도 복합 |
| 랭킹 | `hybridRank()` — 규칙(60%) + 유사도(40%) + PC연계 부스팅 |
| D 평가 | `recommendDetection()` 장비 키워드 직접 산정 |
| 산업DB | `KrIndustryDetection` → 전체 풀 fallback |
