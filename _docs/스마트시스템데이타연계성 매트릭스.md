## 1. ProjectLinkage 통합 DB 구조

### 1.1 ProjectLinkage 테이블 (중앙 집중식 연동 관리)

```prisma
model ProjectLinkage {
  id        String   @id @default(uuid())
  
  // ★★★ 연동 대상들 ★★★
  apqpNo    String?  // 상위 APQP 번호 (pj26-n001)
  pfmeaId   String?  // PFMEA ID (pfm26-n001)
  dfmeaId   String?  // DFMEA ID (dfm26-n001)
  pfdNo     String?  // PFD 번호 (pfd26-n001)
  cpNo      String?  // CP 번호 (cpl26-n001-l01)
  
  // ★★★ 기초정보 (APQP에서 연동, 모든 문서와 공유) ★★★
  companyName         String?  // 회사명
  engineeringLocation String?  // 엔지니어링 위치
  customerName        String?  // 고객명
  modelYear           String?  // 모델연식/플랫폼
  subject             String?  // 프로젝트명
  projectName         String?  // 프로젝트명 (표시용)
  startDate           String?  // 시작일자
  revisionDate        String?  // 개정일자
  processResponsibility String? // 공정책임 부서
  confidentialityLevel  String? // 기밀수준
  responsibleName     String?  // 책임자 성명
  
  // 연동 상태
  linkType    String   @default("auto")
  status      String   @default("active")
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  createdBy   String?
}
```

### 1.2 데이터 흐름 원칙

| 원칙 | 설명 |
|------|------|
| **APQP가 마스터** | 기초정보는 APQP에서 최초 생성, 하위 문서로 자동 전파 |
| **ProjectLinkage가 중앙 저장소** | 모든 연동 문서의 공통 정보는 ProjectLinkage에 저장 |
| **하위 문서는 독립 수정 가능** | 각 문서에서 기초정보 수정 시 해당 문서에만 저장 (독립 운영) |
| **CFT는 각 문서의 고유 데이터** | CFT 멤버는 공유하지 않음 (각 문서별 독립 관리) |

| 시작일자 | ● | ● | ● | ● | ● | APQP → 전체 |
| 개정일자 | ● | ● | ● | ● | ● | APQP → 전체 |
| 공정책임 부서 | - | ○ | ● | ● | ● | PFMEA → CP/PFD |
| 기밀수준 | ● | ● | ● | ● | ● | APQP → 전체 |

| **CFT 멤버** | ● | ● | ● | ● | ● | **공유 안 함 (독립 관리)** |

    };
  }

    docData.companyName = linkageData.data[0].companyName;
    docData.customerName = linkageData.data[0].customerName;
    // ... 기타 필드
  }
  
  return docData;
}

### 6.1 구조 연동 (Structure Sync)

| 원본 | 대상 | 버튼명 | 동작 |
|------|------|--------|------|
| PFD → FMEA | FMEA 구조연동 | PFD 구조로 FMEA 생성 |
| PFD → CP | CP 구조연동 | PFD 구조로 CP 생성 |
| FMEA → CP | CP 구조연동 | FMEA 구조로 CP 생성 |
