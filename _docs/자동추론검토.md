## 2. 삭제 이유

### 2.1 DB 우선 원칙 위반
- ID 패턴을 분석하여 관계를 **추론**하는 것은 실제 DB에 저장된 관계와 불일치할 수 있음
- 예: `pfm26-p001-s`가 있어도 실제로는 `pj26-p001`과 연결되지 않았을 수 있음

### 2.2 신규 모드에서의 부작용
- 신규 등록 시 ID가 생성되면 자동 추론이 동작하여 빈 폼이 아닌 데이터가 채워짐
- 사용자가 새 프로젝트를 생성하려는데 기존 데이터가 로드되는 문제 발생

### 2.3 데이터 무결성 문제
- 추론된 ID가 실제 DB에 존재하지 않을 수 있음
- ProjectLinkage 테이블과 실제 데이터 간의 불일치 가능성

## 3. Smart System에서 자동 추론이 필요한 기능

### 3.1 ✅ 필요한 경우 (DB 저장 & 로드 기반)

| 기능 | 설명 | 구현 방법 |
|------|------|-----------|
| **APQP 생성 시 연동 문서 자동 생성** | APQP 저장 시 연동 PFMEA/DFMEA/CP/PFD ID 자동 생성 | API POST에서 DB에 저장 (현재 구현됨) |
| **편집 모드 데이터 로드** | 기존 프로젝트 편집 시 저장된 관계 데이터 로드 | `loadData`에서 DB API 응답 사용 |
| **ProjectLinkage 테이블 조회** | 프로젝트 간 관계 조회 | DB 쿼리로 정확한 데이터 반환 |

### 3.2 ❌ 불필요한 경우 (클라이언트 추론)

| 기능 | 문제점 | 대안 |
|------|--------|------|
| **ID 패턴 기반 상위 문서 추론** | 실제 관계와 불일치 가능 | DB에서 `parentApqpNo`, `parentFmeaId` 필드 사용 |
| **ID 변경 시 자동 연동 ID 생성** | 신규/편집 모드 구분 어려움 | 저장 시 서버에서 생성 |
| **기초정보 자동 채움** | 수동 입력 덮어쓰기 위험 | 명시적 "연동" 버튼으로 변경 고려 |

## 4. 향후 개선 제안

### 4.1 명시적 연동 기능 추가
```
┌─────────────────────────────────────────────────┐
│  상위 APQP: [pj26-001]  [🔗 기초정보 연동]      │
│                                                 │
│  ※ 버튼 클릭 시 APQP의 기초정보를 가져옵니다   │
└─────────────────────────────────────────────────┘
```

- **자동 연동 대신 수동 트리거**로 변경
- 사용자가 명시적으로 "연동" 버튼을 클릭해야 기초정보가 채워짐
- 이미 입력된 데이터 보호

### 4.2 DB 스키마 개선
```prisma
model PfmeaRegistration {
  fmeaId         String   @id
  parentApqpNo   String?  // ★ 상위 APQP 참조 (DB 저장)
  linkedPfdNos   String[] // ★ 연동 PFD 목록 (DB 저장)
  linkedCpNos    String[] // ★ 연동 CP 목록 (DB 저장)
  // ...
}
```

- 관계 정보를 DB에 명시적으로 저장
- 클라이언트에서 추론하지 않고 DB 필드 사용

### 4.3 데이터 로드 흐름 통일
```
[신규 모드]
  └─ 빈 폼 표시 (ID만 생성)
  └─ 사용자가 상위 APQP 선택 시 → "연동" 버튼으로 정보 가져오기

[편집 모드]
  └─ loadData() 호출 → DB API 응답으로 폼 채우기
  └─ parentApqpNo, linkedPfdNos 등 DB에 저장된 관계 사용
```

## 5. 참고: ID 체계

| 프리픽스 | 모듈 | 예시 |
|----------|------|------|
| `pj` | APQP | pj26-001 |
| `pfm` | PFMEA | pfm26-p001-s |
| `dfm` | DFMEA | dfm26-m001-s |
| `cpl` | CP | cpl26-p001-l01 |
| `pfd` | PFD | pfd26-p001-l01 |

### 접미사 의미
- `-s` : 단독(Solo) 문서
- `-lNN` : 연동 그룹 NN번 (예: -l01, -l02)

## 6. 결론

| 항목 | 이전 방식 | 현재 방식 |
|------|-----------|-----------|
| **데이터 소스** | ID 패턴 추론 + DB 폴백 | DB 데이터만 사용 |
| **신규 모드** | 자동 추론으로 데이터 채움 | 빈 폼 표시 |
| **편집 모드** | 추론 + DB 혼합 | DB API 응답만 사용 |
| **기초정보 연동** | 자동 (useEffect) | 삭제됨 (향후 수동 버튼 고려) |

### 핵심 원칙
> **"추론하지 말고, DB에 저장하고 DB에서 로드하라"**
